# Inference Engine Placement

This document addresses a key question that sits at the intersection of UX, performance, and architecture:

> **When an Orchestrator is deployed remotely, where does local inference run — and can the two be mixed?**

The short answer is: **Yes. It is user choice, and it is technically elegant to implement.**

The Orchestrator and the Inference Engine are **completely decoupled** from each other. They communicate via a well-defined protocol (OpenAI-compatible HTTP API), which means they can run on completely separate machines with no code changes.

---

## The Two Axes of Placement

The user controls two independent knobs:

| Orchestrator Location | Inference Engine Location | Use Case |
|---|---|---|
| **Local (Tauri)** | **Local (same machine)** | The default "all on my Mac" mode. Maximum privacy, zero cloud costs. |
| **Local (Tauri)** | **Remote (calling a cloud LLM)** | The standard cloud-powered mode. Existing OpenAI/Anthropic support. |
| **Remote (VPS/Mac Mini)** | **Remote (same server)** | A fully self-hosted headless agent — privacy-first but hardware-intensive. |
| **Remote (VPS/Mac Mini)** | **Local (user's machine)** | The "hybrid" mode. Agent brains are remote, but inference GPU stays local. |

All four of these are technically possible in ThinClaw's architecture, because of a single principle:

---

## The Key Principle: Local Inference as an OpenAI-Compatible API

The inference engines — whether **MLX, llama.cpp, Ollama, or LM Studio** — all support exposing a local HTTP server that speaks the OpenAI REST API format.

| Engine | Local HTTP Server | Default Port |
|---|---|---|
| MLX (via `mlx_lm.server`) | `http://localhost:8080/v1` | 8080 |
| llama.cpp server | `http://127.0.0.1:8080/v1` | 8080 |
| Ollama | `http://localhost:11434/v1` | 11434 |
| LM Studio | `http://localhost:1234/v1` | 1234 |

This means the Rust Orchestrator treats **every inference endpoint identically**, whether it is:
- `https://api.openai.com/v1` — a commercial cloud API
- `http://localhost:8080/v1` — a local MLX server on the same machine
- `http://192.168.1.50:8080/v1` — a local MLX server on the user's MacBook, called by a remote VPS Orchestrator

In the Rust Orchestrator, this is represented by a single unified provider struct:

```rust
// In rig-core, a "remote" provider and a local inference server
// look and behave identically from the Orchestrator's perspective.

let local_inference = rig::providers::openai::Client::from_url(
    "your-mlx-key",               // local servers usually accept any non-empty key
    "http://localhost:8080/v1",    // <-- the only difference: the base URL
);

let cloud_openai = rig::providers::openai::Client::new("sk-proj-...");
```

## Mode 4 in Detail: Remote Orchestrator → Local Inference

This is the most sophisticated mode. The user's MacBook runs **only** the local inference server (e.g., `mlx_lm.server`) while the Orchestrator runs on a remote VPS.

**The Setup:**
1. User starts the MLX server on their MacBook: `mlx_lm.server --model lmstudio-community/Llama-3.2-3B-Instruct-MLX-8bit --port 8080`
2. The MacBook's MLX server is now reachable at `https://users-macbook.tailscale.net:8080/v1` (via Tailscale, WireGuard, or reverse proxy — see security note below).
3. The user configures the Remote Orchestrator to use `endpoint: "https://users-macbook.tailscale.net:8080"` as its inference provider.
4. The Remote Orchestrator manages memory, tools, context, heartbeats, PII stripping.
5. All generation tokens are generated by the Apple Silicon GPU on the MacBook.

**Practical Benefit:** You get the low latency and zero per-token cost of MLX inference, combined with a 24/7 always-on remote Orchestrator that handles scheduling, tool execution, and channel integration even when the UI is closed.

> ⚠️ **Security Note:** The local inference server must **never** be exposed directly to the public internet. Use a private tunnel (Tailscale mesh VPN is strongly recommended, and handles NAT traversal automatically) or a WireGuard VPN to create a secure private link between the remote VPS and the user's MacBook. The server binds to `localhost` by default for safety; the user must explicitly opt into exposing it — and the Tauri app should communicate this risk clearly.

## How ThinClaw Configures This

In the Rust config (`~/.thinclaw/config.toml`), the provider section is completely agnostic to whether the endpoint is local or remote:

```toml
[inference]
# This single field controls where generation happens.
# Could be OpenAI, a local MLX server, or a remote Mac Mini.
endpoint = "http://localhost:8080/v1"
api_key  = "local"              # MLX/llama.cpp accept any value

# OR — point to a remote Mac Mini over Tailscale
# endpoint = "http://100.64.0.5:8080/v1"

# OR — use a commercial cloud provider
# endpoint = "https://api.openai.com/v1"
# api_key  = "sk-proj-..."
```

The Tauri frontend exposes this as a user-friendly **Provider Settings** UI:

```
[Inference Location]
  ○ Cloud (OpenAI, Anthropic, OpenRouter) — Auto-selects based on configured API keys
  ● Local (MLX / llama.cpp)              — Runs on this machine
  ○ Custom Endpoint                       — Enter URL manually (for Remote Mac Mini / Tailscale etc.)
```

## Engine Selection Based on Remote OS

When the user deploys the Orchestrator to a **remote machine** (a VPS, a Mac Mini, a Raspberry Pi), ThinClaw asks the user to specify the target OS during setup. This drives which local inference sidecar is bundled or recommended:

| Remote OS | Recommended Engine | Sidecar Strategy |
|---|---|---|
| **macOS (Apple Silicon)** | MLX via `mlx_lm.server` | Run via `std::process::Command`, auto-started by Orchestrator on boot. |
| **Linux (NVIDIA GPU)** | llama.cpp with CUDA | Pre-compiled CUDA binary shipped as a sidecar alongside `thinclaw-server`. |
| **Linux (CPU only)** | llama.cpp (CPU build) | Pre-compiled CPU binary — slower but dependency-free. |
| **Windows** | llama.cpp (CPU or CUDA) | Bundled in the ThinClaw Windows installer. |

For the headless server binary (`thinclaw-server`), the `Cargo.toml` uses **conditional compilation features** to avoid cross-platform bloat:

```toml
[features]
default     = []
mlx         = []          # macOS only; managed out-of-process
llama-cpp   = []          # wraps bundled llama.cpp sidecar binary
cuda        = ["llama-cpp"]  # enables CUDA variant of llama.cpp sidecar
```

## Summary: The 2×2 Decision Tree

```
Is the ORCHESTRATOR local or remote?
│
├── LOCAL (Tauri Desktop App)
│    ├── Use local inference (MLX/llama.cpp on same machine)?  → Config: endpoint = "http://localhost:8080"
│    └── Use cloud inference (OpenAI/Anthropic)?              → Config: endpoint = "https://api.openai.com/v1"
│
└── REMOTE (Headless VPS / Mac Mini)
     ├── Deploy inference ON the remote server?
     │    └── Auto-start the correct engine sidecar based on detected OS.
     │        (macOS remote → MLX; Linux remote → llama.cpp with best available backend)
     │
     └── Keep inference LOCAL on the user's machine?
          └── User exposes MLX/llama.cpp server via Tailscale.
              Config on remote: endpoint = "http://100.64.x.x:8080"
              Warning shown in Tauri UI: "Your device must stay online for the agent to generate responses."
```

**The Recommendation for ThinClaw:**
Start with the two simpler modes (All-Local and All-Cloud). Then add the Tailscale-bridged hybrid mode as an advanced/experimental feature once the core is stable, since it requires careful security UX (the user must understand the tradeoffs of exposing any local inference port, even over VPN).
