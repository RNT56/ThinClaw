# Hardware Bridge: Local Sensor Access for Remote Agents

When the Rust Orchestrator is deployed remotely (headless VPS or Mac Mini), it does not have physical access to the user's camera, microphone, or screen. However, the local Tauri Companion App **does**.

The Hardware Bridge is an opt-in system that allows the Remote Orchestrator to request local sensor data via the existing WebSocket connection, with explicit user approval for every request type.

> **Implementation Status (2026-03-01):** The WS RPC design below is the **original specification** for remote-mode usage. For **desktop mode** (IronClaw running in-process inside Scrappy), the architecture has been simplified to an internal Rust trait — see `src/hardware_bridge.rs` which implements `ToolBridge`, `BridgedTool`, and `SessionApprovals`. The WS RPC approach remains valid for future remote orchestrator scenarios but is not currently wired.

---

## 1. The Core Design Principle: Opt-In Per Sensor

The Hardware Bridge is **never auto-enabled**. The user must:
1. Explicitly enable "Hardware Bridge" in the Tauri Remote Settings UI.
2. Grant macOS system permissions for each sensor (Camera, Microphone, Screen Recording) to ThinClaw.
3. Optionally set a **pre-approval trust level** per sensor (`always-ask` / `session-approved` / `trusted-until-revoked`).

If the Remote Orchestrator requests a hardware bridged action and the Bridge is disabled, the Orchestrator receives an `rpc.denied` response immediately and the agent should handle this gracefully (e.g., "I cannot access your camera in this mode").

## 2. Tauri APIs for Each Sensor

| Sensor | macOS API Access (Rust) | Required Permission |
|---|---|---|
| **Camera** | `nokhwa` crate (cross-platform camera capture) | `NSCameraUsageDescription` in `Info.plist` |
| **Microphone** | `cpal` crate (audio capture) | `NSMicrophoneUsageDescription` in `Info.plist` |
| **Screen Recording** | `scrap` crate (cross-platform screen capture) | Screen Recording permission (macOS 10.15+) |
| **Screenshot (single)** | `screenshots` crate (simpler than scrap for stills) | Screen Recording permission |

These are standard Tauri sidecar/plugin capabilities. On macOS, the first time each is invoked, the OS shows its own system permission dialog to the user.

## 3. The RPC Protocol

Hardware Bridge actions use the standard WebSocket message envelope (see `NETWORKING_RS.md`) with these specific types:

### Request (Orchestrator → Tauri App)
```json
{
  "id": "rpc-uuid-1234",
  "type": "tool.rpc.request",
  "payload": {
    "action": "capture_camera_frame",
    "params": {
      "device_index": 0,
      "format": "jpeg",
      "quality": 80
    }
  }
}
```

**Available Actions:**

| Action | Description | Params |
|---|---|---|
| `capture_camera_frame` | Single JPEG/PNG frame from webcam | `device_index`, `format`, `quality` |
| `record_audio_clip` | Record N seconds of microphone audio | `duration_secs`, `format` (opus/wav) |
| `capture_screenshot` | Full screen or window screenshot | `screen_index`, `format` |
| `list_audio_devices` | List available microphone names/IDs | none |
| `list_camera_devices` | List available camera names/IDs | none |

### Approval Step (Tauri App → User)

Before executing any hardware bridge action, the Tauri app shows a native OS-level approval modal:

```
┌─────────────────────────────────────────────────┐
│ 📷  ThinClaw Remote Agent Request               │
│                                                 │
│  Your remote agent (thinclaw-server) is asking  │
│  to access your camera.                         │
│                                                 │
│  Reason: "The user asked me to check if their  │
│  package arrived at the door."                  │
│                                                 │
│  [Deny]  [Allow Once]  [Allow for this session] │
└─────────────────────────────────────────────────┘
```

The Orchestrator **must** include a human-readable `reason` string in the RPC payload (generated by the LLM or hardcoded by the tool). This reason is shown in the approval dialog.

### Response (Tauri App → Orchestrator)

**If Approved:**
```json
{
  "id": "rpc-uuid-1234",
  "type": "tool.rpc.response",
  "payload": {
    "success": true,
    "action": "capture_camera_frame",
    "data": {
      "format": "jpeg",
      "base64": "/9j/4AAQSkZJRgABAQ..."
    }
  }
}
```

**If Denied:**
```json
{
  "id": "rpc-uuid-1234",
  "type": "tool.rpc.response",
  "payload": {
    "success": false,
    "error": "user_denied",
    "message": "The user denied access to the camera."
  }
}
```

## 4. The Rust Tool Definition on the Orchestrator

The Remote Orchestrator registers these bridged tools into the RIG Agent's toolkit. When the LLM calls a bridged tool, the Orchestrator's tool implementation sends the RPC over the WebSocket (instead of executing locally) and waits for the response:

```rust
use rig::{completion::ToolDefinition, tool::Tool};
use tokio::sync::oneshot;

pub struct CameraCaptureTool {
    ws_tx: mpsc::Sender<WsMessage>,           // Send RPC to client
    rpc_registry: Arc<Mutex<HashMap<String, oneshot::Sender<RpcResponse>>>>,
}

#[async_trait]
impl Tool for CameraCaptureTool {
    const NAME: &'static str = "capture_camera_frame";

    async fn call(&self, _params: CameraParams) -> Result<String, ToolError> {
        let rpc_id = uuid::Uuid::new_v4().to_string();
        let (tx, rx) = oneshot::channel::<RpcResponse>();

        // Register expectation
        self.rpc_registry.lock().await.insert(rpc_id.clone(), tx);

        // Fire RPC to Tauri thin client over WebSocket
        self.ws_tx.send(WsMessage::ToolRpc {
            id: rpc_id,
            action: "capture_camera_frame".to_string(),
            reason: "Checking camera feed as requested by user".to_string(),
        }).await?;

        // Wait for the Tauri App to respond (with a timeout)
        match tokio::time::timeout(std::time::Duration::from_secs(30), rx).await {
            Ok(Ok(resp)) if resp.success => {
                // Return the base64 image to the LLM as a multimodal message content
                Ok(resp.data.base64)
            }
            Ok(Ok(resp)) => Err(ToolError::UserDenied(resp.message)),
            Ok(Err(_)) => Err(ToolError::RpcCancelled),
            Err(_) => Err(ToolError::Timeout("Camera RPC timed out after 30s".to_string())),
        }
    }
}
```

The key pattern is **async RPC with a 30-second timeout**. The LLM's tool call blocks until the user approves or denies on their local device.

## 5. Rust Implementation in the Tauri App

The Tauri Companion App's WebSocket handler watches for `tool.rpc.request` messages and dispatches them:

```rust
async fn handle_rpc_request(msg: RpcRequest, app_handle: tauri::AppHandle) -> RpcResponse {
    match msg.action.as_str() {
        "capture_camera_frame" => {
            // 1. Show approval dialog via Tauri dialog plugin
            let approved = show_approval_dialog(
                &app_handle,
                "Camera Access",
                &msg.reason
            ).await;

            if !approved {
                return RpcResponse::denied("user_denied");
            }

            // 2. Capture frame with nokhwa
            capture_camera_frame_nokhwa(msg.params).await
        },

        "record_audio_clip" => {
            let approved = show_approval_dialog(
                &app_handle,
                "Microphone Access",
                &msg.reason
            ).await;

            if !approved { return RpcResponse::denied("user_denied"); }

            record_audio_cpal(msg.params).await
        },

        "capture_screenshot" => {
            // Screenshots require Screen Recording permission
            let approved = show_approval_dialog(
                &app_handle,
                "Screen Recording",
                &msg.reason
            ).await;

            if !approved { return RpcResponse::denied("user_denied"); }

            capture_screenshot_scap(msg.params).await
        },

        _ => RpcResponse::error("unknown_action"),
    }
}
```

## 6. Security Boundaries

| Rule | Detail |
|---|---|
| **Explicit opt-in only** | Bridge is disabled by default. User enables per-session or persistently in Settings. |
| **One-shot or session approval** | User chooses: approve once, approve for this session, or always allow (per action type). |
| **No raw audio streaming** | Audio is always recorded as a fixed-duration clip, not a live stream. Prevents using the device as a covert mic. |
| **No continuous screen capture** | Screenshots are always single frames requested explicitly. No screen recording sessions. |
| **Reason string required** | The LLM/tool must provide a human-readable reason. Empty reason strings are rejected. |
| **30-second hard timeout** | If the user doesn't respond to the approval dialog in 30 seconds, access is automatically denied. |
| **All bridge data is ephemeral** | The Orchestrator receives the image/audio bytes, uses them in the LLM context for that turn, and does not persist them to disk. |

## 7. Tauri `Info.plist` Additions

For macOS, these usage description strings must be added to `src-tauri/Info.plist` so the OS allows the permission prompts:

```xml
<key>NSCameraUsageDescription</key>
<string>ThinClaw uses your camera when your remote AI agent requests to capture a photo with your permission.</string>

<key>NSMicrophoneUsageDescription</key>
<string>ThinClaw uses your microphone when your remote AI agent requests to record an audio clip with your permission.</string>
```

Screen Recording must be granted by the user in **System Settings → Privacy & Security → Screen Recording** and cannot be requested via plist — it requires a manual one-time system permission.

## 8. Crate Summary

Add to `src-tauri/Cargo.toml`:

```toml
[target.'cfg(not(feature = "headless"))'.dependencies]
nokhwa    = { version = "0.10", features = ["input-native"] }   # Camera
cpal      = "0.15"                                               # Microphone
scap      = "0.1"                                                # Screen capture
```

Note: These crates are only compiled into the Tauri Companion App (`not(headless)`). The headless Orchestrator binary never compiles them — it only sends RPCs and receives base64 payload responses.
