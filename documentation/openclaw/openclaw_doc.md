Deep Dive: OpenClaw Agent Architecture and Operation

Agent Construction and Core Loop

Initialization and Base Components: OpenClaw is implemented as a TypeScript CLI application that runs a single, embedded “agent” process on the host machine ￼. Under the hood, it leverages the Pi agent runtime (an open-source coding agent by Mario Zechner) as its core, integrating Pi’s minimal toolset and extension system ￼ ￼. OpenClaw’s agent runtime is essentially derived from Pi’s codebase (“pi-mono”) but with custom session management, discovery, and tool wiring specific to OpenClaw ￼ ￼. When you start the agent (e.g. via openclaw gateway), the system sets up a Gateway server and a single long-lived agent instance. This agent uses a designated workspace directory (by default ~/.openclaw/workspace) as its working directory and memory store ￼. On first run, OpenClaw will generate default “bootstrap” files in the workspace (such as AGENTS.md, SOUL.md, etc.) containing template instructions and persona information for the agent ￼ ￼. These files are loaded at the start of each session and injected into the system prompt to establish the agent’s identity, rules, and user profile.

Core Loop Execution: The heart of OpenClaw is its agent loop, which processes an input message into a series of actions and ultimately a response ￼. The loop runs as a single serialized sequence per session to avoid race conditions. When a user message arrives, the Gateway enqueues an agent command for that session (sessions are identified by user/channel) and immediately responds with an acknowledgment (including a run ID and accept timestamp) ￼. The actual agent execution then proceeds in stages:
	1.	Session Setup: The agent resolves which LLM model to use (based on config or conversation context) and prepares the session. It loads any enabled skills (extended capabilities or plugins) and the latest snapshot of those skills into the runtime environment ￼. The agent also ensures the workspace exists (creating a sandboxed copy if sandbox mode is enabled for isolation) and locks the session transcript for writing ￼. All the relevant bootstrap files (persona, instructions, memory primers) are read and injected into the system prompt at this stage ￼. Large files are truncated to keep the prompt within token limits, and missing files are noted with a placeholder line ￼.
	2.	Prompt Assembly: OpenClaw dynamically builds the system prompt for the LLM. It starts with a base prompt (defining the agent’s fundamental behavior), then appends the content from the workspace files (AGENTS.md, SOUL.md, etc.), any skill-provided instructions, and possibly recent context from long-term memory ￼. This assembled prompt represents the “brain” of the agent for this turn. The conversation history (stored as JSONL transcripts on disk) is also loaded – recent messages from the session are appended so the model knows the dialogue so far ￼. OpenClaw uses a context window guard to ensure the prompt and history fit within the model’s token limit; if near capacity, it will perform a compaction (e.g. summarizing or dropping older content) or gracefully refuse if it cannot proceed ￼.
	3.	LLM Inference and Planning: With the context ready, the agent invokes the configured LLM (e.g. GPT-4, Claude, etc.) via its API. The agent runtime (via Pi core) streams the model’s response tokens back as they arrive ￼. The model’s output may contain either a direct answer or a tool invocation. OpenClaw’s prompt style encourages the model to act as a planner by producing special tokens or JSON indicating tool usage when needed ￼. For example, the model can decide to call a browser tool or run a shell command if the user’s request requires it. This “think-act” loop is iterative: whenever the model outputs a tool call, the agent intercepts it before finalizing the reply.
	4.	Tool Execution Loop: If a tool call is present in the model’s output, the agent enters an execution phase. OpenClaw’s integration with Pi means a set of core tools are always available to the model (these include file read/write, code execution via shell, editing files, etc.) ￼. Additional tools (like web browsing, sending messages, or custom skills) are also registered in the runtime. The agent takes the tool command from the model’s output and invokes the corresponding function on the host machine ￼. For example, a "exec" command might be run in a sandboxed Docker environment on the host, or a "browser.navigate" command might launch a headless browser. Each tool execution produces a result (output text, file content, error message, etc.). The agent captures that result and writes it back into the conversation for the model to consume on the next loop iteration ￼. Essentially, the result is appended to the session transcript (often formatted or summarized) so the LLM can “see” what happened and decide the next step. OpenClaw continues alternating between model inference and tool execution in this loop until the model produces a final answer or a stop condition is reached (e.g. hitting a turn limit around ~20 tool calls or responses by default) ￼. Notably, OpenClaw’s design is serial by default – only one tool runs at a time per agent session – which avoids the chaos of interleaved logs or race conditions that can plague async multi-tool agents ￼. However, low-risk tasks (like certain notifications) could be configured to run in parallel lanes if explicitly allowed, using the underlying queue system.
	5.	Streaming and Responding: As the LLM generates the final answer (i.e. an assistant message not requesting any further tool use), OpenClaw streams this text back to the user. It supports both token-level streaming and “block streaming” – where partial chunks are sent at natural breakpoints (like paragraphs) for a smoother experience ￼. Additionally, OpenClaw can stream the assistant’s reasoning separately if configured (it calls these “thinking” levels) – for example, in a verbose mode the agent might reveal its thought process or tool outputs live in the UI. By default, the agent only sends the user the polished final answer, but the underlying system can include inline tool summaries if verbosity is turned on and the channel supports it ￼. After sending the reply, the agent closes out the run by emitting a lifecycle “end” event along with usage metadata (tokens used, time taken, etc.) ￼. The session transcript (JSONL log) is then updated on disk with the new conversation turn, including all user messages, assistant replies, and tool invocations/results that occurred.

Task Execution & Templates: OpenClaw doesn’t hardcode specific “task scripts” – instead, tasks are dynamically planned by the LLM using the available tools. The workspace markdown files act as templates for behavior and can themselves prompt certain actions. For instance, if a BOOT.md exists (a startup checklist), OpenClaw will instruct the agent to run through those steps when the system (Gateway) starts ￼. Similarly, a HEARTBEAT.md file can list periodic tasks (the agent will execute these on a schedule, e.g. check emails every hour, if heartbeats are enabled) ￼. These are implemented as just special messages injected into the agent’s queue by internal triggers (not separate cron processes), so the agent treats them like user prompts to act on. In summary, the “agent construction” uses a single LLM-driven loop augmented by tools; the agent is initialized with persona and instruction templates, and it iteratively plans and executes tasks using tool calls until it arrives at an answer.

Internal Communication and Message Flow

Gateway and Controller: The central hub of OpenClaw is the Gateway process, which plays the role of a controller and message router. The Gateway runs as a daemon (or foreground process) that listens for incoming messages from various Channel Adapters (messenger-specific integrations) and from UI/CLI clients over WebSocket ￼ ￼. Each chat platform (WhatsApp, Telegram, Slack, etc.) has an adapter that normalizes incoming messages into a standard format for the Gateway ￼. When a message arrives, the Gateway determines which session it belongs to (based on the sender or chat context) and enqueues it into that session’s message lane ￼. OpenClaw uses a lane-based Command Queue to manage concurrency: each session (conversation) gets its own dedicated queue (or “lane”) so that only one agent loop runs at a time per session ￼. This serialization is by design – it ensures that the agent’s memory (transcript) updates and tool actions don’t overlap or conflict within a single session ￼. (There is also an optional global lane to limit total parallelism, and configurable queue modes to handle how new messages are queued during an ongoing agent reply – e.g. accumulate them or inject them mid-stream ￼.)

Inside the Gateway, the incoming message is turned into an agent RPC request (if coming from a user/channel) and handed to the agent runtime ￼. The Gateway’s controller logic (sometimes dubbed the “Agent Runner”) validates the request, loads the correct session state, and spawns the agent loop (via agentCommand in the code) to handle it ￼. The agent runtime then interacts with the Pi-based core as described earlier. The internal components involved include:
	•	Session Manager / Memory – responsible for reading and appending to the JSONL transcript for the session, and for managing long-term memory files. It ensures a write-lock so that only the active run modifies the session log at a time ￼.
	•	Tool/Skill Registry – a component that holds all available tool definitions and skill modules. At startup, OpenClaw loads built-in tools (file read/write, shell exec, web browser, etc.) and any user-installed skills into this registry ￼ ￼. When the LLM requests a tool, the registry maps the call to the actual implementation function and executes it. Tools are invoked via an internal RPC-like mechanism: for example, a model output like {"tool":"exec","args":"ls -la"} would be caught and trigger the Exec tool’s handler in the runtime. Tool execution events are emitted internally so other subsystems (like the UI or logging) know when a tool started, produced output, or ended ￼.
	•	Memory and Search – OpenClaw augments the agent with a simple but effective memory retrieval system. All past conversations are persisted (nothing is dropped unless manually purged), and OpenClaw can perform both semantic vector search (via SQLite with a vector extension) and keyword full-text search (FTS5) over the transcripts ￼ ￼. There is a concept of a Memory Tool or search capability that the agent can use to recall facts: for example, if the agent needs to “remember” something from a previous day, it could query its memory index. In practice, the agent often writes important information to Markdown files (e.g. memory/2026-02-01.md) via the file-write tool, and can later read them. The design is deliberately simple: when a new chat session starts, OpenClaw may automatically summarize the last session into a memory file, but it does not perform complex merging or forgetting – older memories remain available indefinitely ￼. This simplicity makes the memory system explainable and avoids the agent confusing itself with hidden state.

Message Passing and Task Tracking: Within a single agent loop, the flow of messages between components is orchestrated by events. The Gateway subscribes to agent events from the runtime: for instance, as the agent calls tools and generates outputs, the runtime fires events like stream: "tool" (with interim tool output), stream: "assistant" (with partial LLM reply deltas), and lifecycle events for start/end ￼ ￼. The Gateway in turn dispatches these to any connected clients (such as the Web UI or other interfaces subscribed to that session’s feed). Internally, if the agent is managing multiple subtasks or even multiple agents, it relies on the structured message log to keep track. OpenClaw supports a notion of sub-agents (spawned agents) and even a multi-agent collaboration via a feature called ClawHub, which can route tasks between agents ￼. For example, one agent can delegate a sub-task to another “sub-agent” (perhaps with a different skill set or persona), and ClawHub handles their coordination by creating a separate session lane for the sub-agent and then relaying results back. Communication between multiple agents or between an agent and a remote node is done through the same Gateway messaging system – effectively all messages are JSON payloads on the WebSocket bus with a specified target session or node. This design keeps multi-agent workflows consistent with single-agent flows, just utilizing multiple lanes (one per agent).

Controller vs Planner vs Executor: OpenClaw does not rigidly separate “planner” and “executor” agents as some frameworks do; instead the LLM itself plans actions, and the agent runtime executes them. The Gateway/Controller ensures that these steps happen in order and under safe conditions. For instance, before running a shell command, the controller checks the execution allowlist and sandbox policies to prevent dangerous operations. OpenClaw employs a safety layer akin to Claude’s code sandbox: there is an exec-approvals.json that records user-approved command patterns for the agent ￼ ￼. If the agent attempts a new or risky command, the system can require explicit approval (or block it outright if it matches known bad patterns). For example, any command with command substitution ($(...)), redirection to system files, or destructive actions like rm -rf / is automatically blocked before execution ￼. This enforcement is handled by the execution subsystem (executor), but at a higher level the agent’s “controller” (Gateway) will catch these and insert an error result into the conversation (so the LLM knows the command failed for safety reasons). In summary, the internal message flow ensures that the Memory component provides context to the Planner (LLM), the planner’s intents are carried out by the Executor (tools), and the overall Controller (Gateway) coordinates these pieces – feeding the LLM new inputs (tool results or user follow-ups) and maintaining ordered, safe operation of the agent.

Web UI Communication and User Interaction

Web UI Overview: OpenClaw includes a built-in Web Control UI (sometimes just called “Control UI”) that allows users to interact with the agent through a browser. By default, the Gateway serves this web interface on the same port as its WebSocket API (port 18789 by default) – for example, you can open http://localhost:18789/ to load the UI ￼ ￼. The Control UI is a static single-page app that communicates with the backend purely via WebSocket to the Gateway’s API ￼. In other words, once loaded in your browser, the UI will initiate a WebSocket connection to ws://localhost:18789 (or a tunnel/proxied address if remote) and perform the handshake just like any other client. The first message sent is a connect frame including an auth token (if the Gateway is configured to require one) ￼. Upon successful connect, the Gateway replies with a hello-ok that contains a snapshot of the agent’s state (presence, health, etc.) ￼. From then on, the Web UI can send user messages and receive the agent’s responses and event streams in real-time.

User Message Flow (WebChat): The Web UI essentially provides a chat interface (often referred to as “WebChat” in the docs). When the user types a message and hits send, the UI issues a WebSocket request of type req:agent with the message and session info ￼. The Gateway treats this identical to a message coming from any other channel – it routes it to the appropriate session lane and returns a response indicating the request was accepted ￼. Almost immediately, the user’s message appears in the UI chat log (locally echoed), and the agent’s typing indicator or partial response will start streaming in as the LLM produces output. The Web UI subscribes to the agent’s event streams by listening for event:agent frames on the WebSocket ￼. As partial assistant events come in (streaming tokens), the UI updates the chat transcript. When a tool event comes in, the Control UI can display the tool’s action or output in a special format (for instance, showing that the agent is running a command or opening a webpage). Finally, when the lifecycle end event arrives marking completion, the UI knows the turn is done and may present any summary or usage stats.

Visible Messages and Logs: One powerful aspect of the Control UI is that it exposes different levels of detail depending on the context (this addresses which messages, tasks, and logs are visible to whom). In a typical messaging app integration (say via WhatsApp or Slack), the end-user only sees the human user messages and the assistant’s final answers – they do not see the agent’s “thoughts” or raw tool outputs unless the agent chooses to explain them. However, in the Web Control UI (which is generally used by the agent’s operator or the tech-savvy user themselves), you can see a more verbose feed. The UI can show internal events like tool execution steps, error tracebacks, and reasoning, because it directly receives the tool and assistant delta events from the Gateway. For example, if the agent decides to call the browser tool to navigate to a page, the Control UI might show a line like “Agent is browsing: opening example.com…” and then perhaps the text snapshot of the page if the agent read it. It streams the tool output via the event interface so you can watch the agent’s actions live ￼. These low-level logs are visible in the Control UI but would not be sent as messages to an ordinary chat channel (to avoid confusing end-users). Similarly, if the agent encounters an error (say a Python exception in a code tool), the stacktrace would be available in the UI logs, whereas the agent might only summarize “I hit an error” to the end-user. This separation ensures that power-users/developers can monitor and debug the agent in the UI, while regular users only get the high-level interaction.

Roles and Sessions in UI: Within the Web UI chat, messages are labeled by role – e.g. “User” vs “Assistant” – to distinguish human inputs from agent responses. OpenClaw uses these roles internally in the JSONL transcript too (each entry has a role field). The UI also has the concept of different contexts or sessions. By default, there is a main private session (often named “main” or “personal assistant”) where the agent’s full capabilities and memory are available. The UI might allow switching to other sessions or group contexts if configured (for instance, if you have the agent linked to a group chat or if you spawn a sub-agent, those could appear as separate conversation threads in the UI). Each session will have its own message log and possibly different visibility of memory (OpenClaw, for example, will not load the global MEMORY.md file when the agent is in a shared group session, to avoid leaking private info ￼). The UI honors these settings, ensuring that what’s shown to the user aligns with their role and context.

Web UI Controls: Beyond just chat, the Control UI provides management panels – for instance, viewing and editing configuration (with a raw JSON editor), monitoring active tools or processes, and managing approvals. The UI’s “Agents” or “Settings” section will show which tools are enabled, allow the user to approve or deny pending exec commands, and display the content of memory files or logs. These features are protected by authentication (only the trusted operator who has the token or local access can use the Control UI). In summary, the Web UI is both a chat frontend and a control dashboard, communicating entirely via the Gateway’s WebSocket API. It differentiates between the assistant’s messages, user’s messages, and system-level events (tools, errors, etc.) and displays each appropriately: end-users in chat get a clean conversation, whereas the operator in Control UI can toggle verbose mode to see the agent’s internal decision-making and task progression live.

Templates and Tools System

Bootstrap Templates: OpenClaw relies on a set of Markdown templates in the agent’s workspace to shape the agent’s personality, knowledge, and behavior. These include files like AGENTS.md (instructions on how to operate and use its memory), SOUL.md (the agent’s persona and boundaries), IDENTITY.md (name and fun details like an emoji avatar), USER.md (information about the user and how to address them), and TOOLS.md (notes on tool conventions) ￼ ￼. During agent startup or the first message of a new session, the contents of these files are injected into the system prompt that the LLM sees ￼. In effect, they serve as a persistent system prompt across conversations, allowing the user to edit them to steer the agent. For example, you might add rules in AGENTS.md like “Always double-check with me before sending an email.” The agent will then carry that instruction into every session. These templates are not static, one-off prompts – they are re-read on each session start, so you can update the agent’s behavior by editing the files. OpenClaw provides default versions (often stored as AGENTS.default.md in the source) which the onboarding process copies into your workspace, but you are free to customize them.

Task Templates (Checklists): Some template files correspond to automated tasks:
	•	BOOTSTRAP.md: A one-time ritual that runs on a brand new workspace. The agent will execute the steps or dialog in this file just once (the idea is to let the agent set itself up, perhaps by introducing itself or creating initial memory structures). After running, the file is meant to be deleted ￼.
	•	BOOT.md: A startup checklist that runs every time the Gateway/agent restarts (if internal hooks are enabled). This could include tasks the agent should do on boot, like greeting the user or checking for updates ￼.
	•	HEARTBEAT.md: A small checklist of tasks that run on a schedule (the “heartbeat”). This is for periodic maintenance or automation – e.g. every 15 minutes the agent could check HEARTBEAT.md and execute, say, “remind me to drink water” or update some status ￼. To conserve tokens, this file is kept very short and is only used if heartbeat is enabled.
These template-driven tasks are implemented by injecting the file content as a system message or instruction at the appropriate times, effectively prompting the agent to carry them out. In addition, OpenClaw supports user-defined scripts or automations (via the Hooks system) to schedule tasks, but those go beyond static templates (they can trigger agent runs programmatically on certain events) ￼ ￼.

Tool Configuration and Invocation: Tools are a first-class concept in OpenClaw. At runtime, the agent has a catalog of tools it can use, each with a name and function. Core tools are always enabled – these include:
	•	File I/O tools: read (to read file contents), write (to create or append to files), edit (to modify files), etc.
	•	Shell execution: exec tool, which runs shell commands. By default this runs inside a Docker sandbox for safety ￼, although it can be configured to run on the host (or even on remote machines via SSH) if elevated mode is enabled.
	•	Browser automation: a browser tool powered by Playwright, which the agent can use to fetch web pages or click elements. Instead of screenshots, OpenClaw’s browser tool returns a semantic DOM snapshot – a text listing of page elements (buttons, links, text) with reference numbers ￼. The agent can then issue follow-up browser actions by referring to these element IDs (e.g. “click the button [ref=1]”). This approach is far more token-efficient and reliable than raw HTML or image data, enabling the agent to navigate websites in an explainable manner ￼ ￼.
	•	Communication tools: e.g. sending messages via SMS or email if those integrations (skills) are added, or posting to chat channels (the agent can act in group chats).
	•	Process management: The agent can spawn background processes or manage long-running commands via specialized tools (OpenClaw has a process tool and supports async background exec tasks) ￼.

Tools are registered with the agent either as built-ins or via Skills/Plugins. Skills are essentially pluggable modules (TypeScript code or config) that can introduce new tools or abilities to the agent ￼. On startup, OpenClaw loads any bundled skills (provided by the project), any user-installed skills in ~/.openclaw/skills, and any workspace-specific skills (in the workspace skills/ directory) ￼. This loading creates a unified list of tools the agent can use, and the agent’s system prompt is automatically augmented to inform the LLM about these tools. For example, the system prompt might include a brief description of each tool and how to call it (this is done in Pi’s minimal style – often the prompt is like: “You can respond with a JSON: {tool: <name>, args: <...>} to use a tool.”). Once the conversation is running, the LLM will output the tool JSON (or a special token format) when it decides to invoke a tool. The runtime captures that and calls the corresponding TypeScript function implementing the tool.

Execution Patterns and Safety: When a tool is invoked, OpenClaw emits tool start and tool end events, and streams any intermediate output. For instance, if the agent runs a shell command via exec, the stdout/stderr from that command is captured line by line. The system will pass through a filter to prevent excessively long output or binary data from flooding the model’s context. Typically, only a summarized or truncated result is fed back into the LLM (with large logs saved to disk if needed). Before logging or emitting tool results, OpenClaw sanitizes them – removing or shortening overly large content, and handling image data (if a tool returned an image, the agent might instead get a placeholder or an uploaded link) ￼ ￼. This ensures the conversation history doesn’t blow up in token size due to tool outputs.

A crucial aspect is the tool policy and approvals. By default, OpenClaw whitelists certain safe commands for the exec tool (like basic ls, grep, ping etc.) and blocks dangerous patterns outright ￼. For anything else, it uses a one-time approval mechanism: when the agent first tries to run a new command, the Gateway will pause and request approval (if running in interactive mode). The user (through the Control UI’s Approvals panel or via CLI) can then allow or deny that command. Approved commands (with their patterns) get added to ~/.openclaw/exec-approvals.json for future reference ￼. This file effectively maps agent IDs to allowed command regex patterns, with timestamps of last use. It gives the user fine-grained control over what the agent can do on their system, preventing the agent from ever running something truly destructive unless explicitly permitted. OpenClaw’s philosophy here is “maximum autonomy within boundaries” ￼ – empower the agent to use powerful tools, but fence off what it shouldn’t do using both automated checks and user governance.

Tool Use in Conversations: From the user’s perspective, tools are used transparently. The agent will usually narrate the outcome of tool use. For example, if the user asks “What’s my system uptime?”, the agent might internally call the exec tool with uptime and then get the result “up 5 days, 4:20” and reply to the user, “Your system has been up for 5 days and 4 hours.” The user might not even realize a tool was invoked. However, if the user is watching in the Control UI with verbose mode, they would see something like:
	•	Assistant → Tool: {tool: "exec", args: "uptime"}
	•	Tool output event: 14:20:01 up 5 days, 4:20, 3 users, load average: ... (truncated)
	•	Assistant final reply: “Your system has been running for about 5 days 4 hours.”

OpenClaw strives to make tool use seamless: the templates in TOOLS.md can contain guidelines for the agent on how to format tool outputs or how verbosely to explain them to the user ￼ (though this file is just advisory and does not directly toggle tool availability).

Another interesting tool is the apply_patch tool ￼. This allows the agent to generate a unified diff (patch) to modify a file, and then apply it automatically. It’s gated behind a config (tools.exec.applyPatch) because of its power. When enabled, the agent can, for example, write code by reading a file, formulating a diff, and applying it – a safer approach than letting it free-form edit files.

In summary, OpenClaw’s tool system is robust and tightly integrated: tools are exposed to the agent via the prompt, invoked through structured outputs, and executed by the runtime with safety checks. Templates like TOOLS.md and the system prompt ensure the agent understands what tools do. The execution patterns (one tool at a time, with streaming outputs) and approval mechanism strike a balance between letting the agent perform complex tasks (browsing, coding, system control) and maintaining user oversight.

Authentication and Role-Based Visibility

User Roles and Access Control: OpenClaw is primarily a personal assistant running on a user’s own machine, so it doesn’t implement multi-user roles in a traditional webapp sense. However, it does enforce distinctions between different connection roles and contexts:
	•	Operator (Owner): This is essentially the person running OpenClaw. By default, the Web Control UI and CLI assume you are the trusted operator. The Gateway can be configured with an authentication token (OPENCLAW_GATEWAY_TOKEN) that must be presented by any client (UI or CLI) connecting to the WebSocket API ￼. This prevents unauthorized access to the agent’s control interface. The operator uses this token (or local network access, if tokenless on localhost) to get full control of the agent. Once connected, the operator has visibility into all sessions, all tools, and can edit configuration and memory. In the UI, there is no further role distinction – it’s assumed only the rightful user has access, and thus they can see everything the agent is doing (including internal thoughts, logs, etc., as described). The operator can also pair new devices and approve/deny tool requests.
	•	Node (Device): Nodes are companion devices (e.g. a mobile app or another computer) that connect to the Gateway with role: "node" in the WebSocket handshake ￼ ￼. These are typically used for things like running a camera or microphone feed, or executing certain commands on a specific device. Each node must go through a pairing process – when a new device connects, the Gateway can require a pairing code or manual approval in the Control UI’s “Pairing” section ￼. Once paired, the Gateway issues a device token so that subsequent connections from that device are trusted ￼. Nodes have restricted capabilities; they advertise what commands or sensors they provide, and they cannot subscribe to arbitrary events or perform chat sends unless permitted. In effect, a node role is a limited agent that acts on behalf of the main agent for certain tasks (for example, an iOS node might allow the agent to fetch GPS location or show a notification on your phone). The pairing and token system ensures only authorized devices can join the agent’s network.
	•	Chat User: This refers to people who interact with the agent via linked chat platforms (WhatsApp, Telegram, etc.). OpenClaw uses channel-specific allowlists to gate who can actually reach the agent. For instance, you can set channels.whatsapp.allowFrom to your own number or a set of IDs ￼. The agent will ignore or politely refuse messages from senders not on the allowed list. This is a basic security measure to prevent random individuals from talking to your personal agent. Users on allowed channels essentially have the ability to send queries and receive answers, but they do not have any control beyond what the agent’s persona permits. They cannot see the agent’s internal chain-of-thought or command it to do privileged operations unless the agent itself is willing to do so. In group chats, the agent can operate but will omit personal or sensitive data (for example, it won’t load MEMORY.md in a group context, as noted, to avoid spilling private info) ￼. Also, certain tools (like file system access) might be configured to be disabled in shared chats for safety.

Role-Based Visibility: The concept of “visibility” in OpenClaw largely comes down to session context and interface rather than formal user roles. The assistant (AI agent) always “sees” everything in its context: system instructions, memory files, tool results, etc. The end-user (over a messaging channel) only sees what the assistant chooses to communicate. The operator (via Control UI or logs) can see both sides – they see the raw conversation including system and tool messages. For example:
	•	The agent might have a system rule (from SOUL.md) saying it should not reveal certain API keys. The user on WhatsApp will never see that rule, but the operator could open SOUL.md or the session transcript and verify that the rule was present.
	•	If the agent uses a tool to fetch some data and then summarizes it for the user, the user sees only the summary. The operator could look at the tool’s actual output in the UI’s log to verify the agent’s summary was accurate.
	•	If two users were hypothetically using the same agent (say in a group), the system can designate one as the “primary user” and others as secondary – the agent might address them differently based on USER.md (which typically is written for the owner). However, widespread multi-user support is not the primary use-case of OpenClaw; it’s more of a single-user assistant, with exceptions for group chats by the owner’s choice.

OpenClaw does allow multiple clients to connect to the Gateway (for example, you could have the Control UI open in a browser and also connect via the CLI or a mobile app). Each client must authenticate (via token or local trust). The Gateway doesn’t yet implement granular RBAC (role-based access control) like admin vs read-only user – it assumes if you’re connected, you’re authorized to use all features. However, security is maintained by limiting who can connect in the first place (loopback-only by default, or requiring the secret token).

Session Visibility Settings: There are a few configuration toggles that affect what the assistant will expose:
	•	Verbose / Thinking Mode: The assistant can be instructed (by a command or config) to operate in a more verbose mode where it shares its reasoning. In the Web UI, there’s often a toggle for “thinking level” which, when high, effectively tells the agent to include its chain-of-thought as part of its response (or as separate messages). The agent won’t do this unless asked (to avoid confusing normal users), but the operator can set a high thinking level in the UI for debugging. Under the hood, this is implemented by adjusting the system prompt to let the model “speak its thoughts” in the answer.
	•	Session Tools Visibility: OpenClaw has a concept called session tools visibility (config key sessionToolsVisibility) which can be “spawned” or “all”. This controls whether tool-use messages are only visible in spawned sub-sessions or in all sessions ￼. This is an advanced setting usually related to multi-agent scenarios (if one agent spawns another to do a task, you might not want the tool calls of the sub-agent to clutter the main session’s output). By default, the user won’t see those internal tool messages unless we’re in the developer UI.

Authentication Summary: In practice, when you first open the Control UI, if an auth token is set, you’ll be prompted for it (the UI will send it as connect.params.auth.token on the handshake) ￼. If the token matches, you’re in and effectively “admin”. Additionally, when a new device tries to connect as a node, the UI will show a pairing request – the user (operator) must approve it, often by comparing a code. Once paired, that device gets a stored identity so it doesn’t need to re-pair each time ￼. All these measures ensure that the agent’s powerful capabilities (like running tools on your machine or accessing personal data) are only ever at the disposal of the person who set it up, and not an arbitrary remote user.

Finally, OpenClaw logs all session activities (in ~/.openclaw/agents/<id>/sessions/ JSONL files) ￼. These logs are only readable by the operator, as they reside on the host filesystem. If one were to share an agent publicly (not common with OpenClaw, but hypothetically), they would need to scrub these transcripts or use the allowFrom settings to restrict usage. As a personal assistant, role-based permissions are mostly about separating the agent’s internal knowledge vs what it outputs, and securing access to the agent’s control interfaces – both of which OpenClaw achieves through the strategies discussed (prompt design and gateway authentication).

In summary, OpenClaw’s architecture emphasizes a single trustworthy operator model with full visibility and control, while end-users (which in most cases is the same operator via a chat interface) get a polished assistant experience. The agent’s internal messages and tasks are visible in the dev UI for debugging but hidden from normal chat interactions. Safety nets like pairing approvals, token auth, and tool allowlists form the backbone of OpenClaw’s role-based access and visibility control ￼ ￼.
