Design of the OpenClaw Default Web UI

Overview: OpenClaw’s default Web UI is a browser-based single-page application (built with Vite and Lit) that serves as a comprehensive control dashboard for your personal AI assistant ￼. The UI is served directly by the OpenClaw gateway on the same port as its WebSocket API (default http://localhost:18789/) ￼. All interactions between the UI and the AI system occur over this WebSocket connection using a JSON message protocol – the UI does not run any AI logic itself, but instead sends commands/requests to the gateway and receives events/responses in real time ￼ ￼. During the initial WebSocket handshake, the UI authenticates as an “operator” client by providing the gateway token or password (set up during onboarding) ￼ ￼. This token-based auth is enforced for security, since the Web UI is an admin interface with powerful controls ￼. Once connected, the Web UI obtains full control-plane access (with appropriate scopes/permissions) to orchestrate the AI assistant.

Architecture & Communication: The Web UI and gateway communicate via a structured WebSocket protocol. On connect, the UI sends a connect request declaring its role (operator) and auth token ￼ ￼. The gateway responds with a confirmation (hello-ok) if protocols and auth match ￼. All subsequent interactions use JSON messages: requests (e.g. calling methods like chat.send or config.get), responses, and events (real-time updates from gateway) ￼. For example, when you send a chat message in the UI, the front-end issues a chat.send request. The gateway immediately responds with an acknowledgment (including a run ID) and then streams the AI’s answer back as a series of chat event messages ￼. This streaming design allows the UI to display the assistant’s reply word-by-word. Many UI actions correspond to specific gateway methods/events in this way. Notably, side-effecting requests (like sending a message) include idempotency keys to avoid duplicate executions on reconnects ￼. All data (conversation history, session state, config, etc.) lives on the gateway – the Web UI is essentially a real-time window into the gateway’s state and operations ￼.

Key Features and Interface Sections

OpenClaw’s default Web UI is feature-rich, providing full visibility and control over your AI assistant. Its interface is divided into multiple sections (tabs/panels), each harnessing gateway APIs to manage different aspects of the system:
	•	Chat Interface (WebChat): The Web UI includes a live chat panel for conversing with your AI agent in real time ￼. You can type messages or commands, send them, and see the assistant’s streamed responses appear in the chat. This chat supports special slash-commands like /stop to abort a running task or /new to start a new session (i.e. reset the conversation context) ￼. A dedicated “Stop” button is provided in the UI during an active AI generation, which triggers a chat.abort request to halt the AI’s response mid-stream ￼. The UI’s chat panel also streams tool usage events: if the agent invokes tools or plugins (e.g. web browser, code executor), you will see live “tool output” cards or messages in the chat reflecting those actions ￼. In effect, the chat interface mirrors what you would experience in a messaging app with the assistant, including real-time typing indicators and incremental message streaming.
	•	Multiple Sessions Management: OpenClaw supports multiple parallel conversation sessions, and the Web UI provides controls to manage these. You can have the assistant maintain separate contexts (for example, one for personal tasks and another for coding help). The UI allows starting a new chat session at any time (via a “New session” button or the /new command) which instructs the gateway to mint a fresh session ID and context ￼ ￼. A dropdown or label (as shown in the screenshot) displays the current session identifier (typically formatted as agent:<agentId>:<sessionKey> – e.g. agent:main:main for the default agent’s primary session) and allows switching if multiple sessions exist. There is also a dedicated “Sessions” panel in the sidebar where all session records are listed ￼. This panel queries the gateway’s session store (the gateway holds session metadata in memory and in ~/.openclaw/agents/<id>/sessions/ files) and shows each session’s details like its display name, associated channel (if any), last activity time, and token usage statistics ￼ ￼. (For instance, group chat sessions might display the group name, whereas a direct chat might be labeled “main”.) The UI refreshes the sessions list automatically when sessions are created or reset via chat commands ￼. By design, the gateway is the source of truth for session data, so the Web UI always fetches session info via the gateway API (sessions.list) rather than tracking state itself ￼. This ensures that token counts (e.g. input/output token usage per session) and session IDs are always accurate and up-to-date as reported by the backend ￼. In summary, the Web UI makes it easy to start new conversations, toggle between sessions, or inspect session histories – all without losing context in each separate thread.
	•	Overview Dashboard: The Overview section (under “Control”) gives a high-level status of your OpenClaw system. It aggregates health info and important notices. For example, it may show which AI model is currently in use, whether the gateway and any connected nodes are healthy (a green “Health OK” indicator is shown in the top bar)【45†】, and summary stats like number of messages processed or uptime. This section uses gateway calls like status and health to snapshot the system state ￼. It’s essentially a quick dashboard to confirm everything is running smoothly.
	•	Channels Integration: In the Channels panel, you can manage and monitor the messaging platforms connected to your OpenClaw (WhatsApp, Telegram, Discord, Slack, etc.). OpenClaw lets you converse with your assistant via these external chat apps, and the Web UI helps configure that. The Channels screen will display each chat channel’s status (e.g. “connected” or awaiting login) and allow certain actions like logging in or scanning a QR code for WhatsApp. For instance, if you have enabled the WhatsApp or Telegram integration, the UI can show a “Login via QR” prompt – clicking it triggers the gateway to generate a login QR code (via web.login.whatsapp etc.), which the UI displays for you to scan ￼. Similarly, channel-specific settings can be tweaked through this interface (under the hood, the UI calls channels.status to get info and config.patch or specific web.login.* methods to perform actions) ￼. This panel essentially bridges the gateway’s channel plugins with a user-friendly interface – you can see which accounts/bots are linked and initiate pairing or re-authentication flows without using the CLI.
	•	Instances (Presence): OpenClaw supports multiple instances or nodes – for example, you might have a Mac and a phone both connected as “nodes” providing additional capabilities (camera, microphone, etc.) or running parallel agents. The Instances section lists all devices currently connected to the gateway (including the main host and any remote nodes) ￼. Each entry shows a device’s ID and the roles/scopes it has (operator or node) ￼ ￼. The UI calls system-presence to get this list ￼. This allows you to verify that, say, your mobile node is online and discoverable. If needed, a “refresh” action can be triggered to update the presence list. In effect, the Instances page gives a quick overview of all active connections to the gateway, ensuring your personal AI network (gateway plus companion devices) is set up correctly.
	•	Cron Jobs: The Web UI also provides a Cron Jobs interface for automation tasks. OpenClaw can run scheduled jobs (background tasks or reminders configured to trigger at certain times or intervals), and the Cron panel lets you manage these visually. It leverages the gateway’s cron APIs (cron.list, cron.add, cron.run, etc.) ￼. In this panel, you can view scheduled jobs, including their schedule, next run time, and status (enabled/disabled). You can manually trigger a run of a cron job on-demand, pause or resume jobs, or create new ones by specifying the command/skill to run and the schedule (e.g. via a form). A history of recent runs might also be shown for each job (with any output or errors), using data from cron.history. This feature essentially turns OpenClaw into an automation hub, and the UI gives non-technical users a way to harness that without editing config files.
	•	Skills and Plugins: OpenClaw’s extensibility comes from its skills (also called plugins or tools), which are modular scripts or integrations that the AI agent can use (for example, sending emails, controlling IoT devices, running web searches, etc.). The Skills section of the Web UI lists all installed skills and their current status ￼. For each skill, you might see whether it’s active, needs an API key, or has any issues. The UI can toggle skills on or off (skills.enable/disable via the gateway) ￼, install new community skills (perhaps via a “ClawHub” skill store link), or update API keys for skills that require authentication. For instance, if you have a “Gmail” skill, the UI might present a field to paste your Gmail API key or OAuth token, which then calls skills.updateKey (a specific action under skills.*) to save it. Essentially, this panel is a graphical plugin manager for your AI: you can review what tools the AI has available and configure them. This encourages a no-code approach to extending capabilities, since new skills can be enabled with a click instead of manual installation. (Under the hood, the gateway’s skills subsystem handles downloading skill code and storing keys, and the UI provides the front-end to invoke those functions.)
	•	Nodes (Device Capabilities): The Nodes panel relates to Instances but drills down into capabilities of each node. If you select a particular node (say your smartphone running the OpenClaw iOS app), the UI can show what capabilities it has advertised – e.g. camera, microphone, screen capture, location, voice etc. ￼ ￼. Nodes declare these at connect time (called “caps” and “commands”), and the gateway shares this info with operator clients ￼ ￼. The UI’s Nodes section likely lists each connected node and the features it provides. You might be able to grant or revoke certain permissions here as well (for example, toggle whether the AI is allowed to use your camera), corresponding to gateway methods like adjusting the exec.approvals or node permission settings ￼. In short, this panel gives you oversight and control over any auxiliary devices in the OpenClaw network, ensuring the AI only uses the hardware capabilities you’re comfortable with.
	•	Config Editor: Because OpenClaw is highly configurable, the Web UI includes a Configuration section (under Settings) that lets you view and modify the settings in your ~/.openclaw/openclaw.json config file through a user-friendly form ￼. On opening this panel, the UI fetches the current config via config.get and also retrieves a JSON schema of all available configuration options (config.schema) ￼. Using that schema, the UI renders a structured form with fields and descriptions for each setting (for example: gateway bind address, default model provider, enabled channels, etc.). You can edit values directly in the form or toggle advanced options. The UI also provides a raw JSON editor mode in case you prefer editing the config file textually ￼. Critically, when you submit changes, the Web UI will call config.set or config.patch to apply them, but it does not immediately overwrite the live config. Instead, it uses config.apply which triggers the gateway to validate and apply the new config atomically ￼. The gateway will check the changes against the base hash (to avoid clobbering any concurrent edits) and then restart/reload with the new settings if validation passes ￼. The UI will reflect success or any validation errors back to you. This mechanism makes it safe to tweak your AI’s configuration from the browser. Common tasks here include switching the default AI model or provider, turning features on/off, or updating keys (all of which can be done without restarting the entire system manually – the UI and gateway coordinate the hot-reload). The config panel thus empowers users to fine-tune OpenClaw through an intuitive interface rather than editing JSON in a text editor.
	•	Debug & Diagnostics: The Debug section is meant for advanced monitoring and troubleshooting. It surfaces low-level information such as system status, model runtime details, event logs, and even allows manual RPC invocations. Specifically, the debug panel can call endpoints like status, health, and models.list to show snapshots of the AI’s state ￼. For example, it may display the current health check results (ensuring all subsystems are OK), list loaded AI models or their current memory usage, and show recent error traces or warnings. A particularly powerful feature here is the live event log: the UI can subscribe to and display incoming events from the gateway (such as tool invocations, errors, or other internal debug info) in real time. There is also likely a facility for manual RPC – you could, for instance, send a crafted JSON request to the gateway from this panel for testing purposes (the docs hint at “manual RPC calls” being possible) ￼. This is useful for developers or power users who want to poke at the system’s internals or test certain behaviors. Essentially, the Debug console turns the Web UI into a mini “dashboard” akin to a developer console, where you can watch the AI’s decisions and internal events live and ensure everything is functioning as expected.
	•	Logs Viewer: In addition to the real-time debug feed, the Web UI offers a Logs section that tails the gateway’s log files in your browser ￼. OpenClaw’s gateway writes logs (to /tmp/openclaw/openclaw-*.log by default), and through the UI you can stream these logs, filter them by level or module, and even export snippets. The UI calls logs.tail to get a live feed of log lines ￼. This is incredibly helpful for tracking what the AI did, especially if something went wrong – you can see error stacks, warnings, or important info without digging through the filesystem. The log viewer likely has controls to pause/resume streaming and to download the log. By integrating logging into the UI, OpenClaw makes debugging easier for users who may not be comfortable SSH’ing into a server or opening terminal windows to view logs. It’s all visible in the web dashboard.
	•	Software Updates: The Web UI even supports updating the OpenClaw installation. In the Settings or Overview area, there is an Update button/feature that triggers the gateway’s self-update process ￼. Internally, this invokes update.run on the gateway, which will pull the latest version (if using a git or package install) and then restart the system ￼. The UI will typically show progress of the update (downloading, installing dependencies, etc.) and might display a restart countdown or a post-update report. This way, you can upgrade to new versions of OpenClaw from your browser dashboard with one click, rather than manually running commands. (After an update + restart, the Web UI would reconnect to the gateway automatically if the token remains the same, thanks to the token stored in local storage.)

In sum, the default Web UI essentially exposes all major controls of OpenClaw’s CLI and APIs in a graphical form ￼ ￼. From chatting with your assistant, to linking it with external chat apps, scheduling tasks, installing plugins, monitoring devices, tweaking settings, and reviewing logs – the Web UI is a one-stop control center. This rich feature set is why the user can fully run and manage their personal AI assistant from a web browser.

Gateway Integration and Data Flow

A core principle of OpenClaw’s Web UI design is that the Gateway is the single source of truth for all data and state. The UI is thin, stateless beyond the current session, and fetches everything on the fly via the WebSocket API. For example, the UI does not store conversation history in the browser; when you open a session or the chat tab, it will call chat.history to retrieve messages from the gateway’s session log ￼. Similarly, session lists, skill lists, etc. are all retrieved via method calls like sessions.list, skills.list, etc., rather than the UI maintaining its own cache ￼. This design ensures that whether the UI is running on the same machine or remotely, it always reflects the live state of whatever gateway it’s connected to. All data sharing happens through structured payloads over the WebSocket – for instance, when a new message arrives, the gateway broadcasts an event {"type":"event","event":"chat","payload":{...}} which the UI listens for and then renders as a new chat bubble in the interface ￼. If a setting is changed via UI, the UI sends a request and the gateway’s response event confirms the change, which the UI then uses to update the display.

WebSocket Role & Security: The Web UI connects as an “operator” client, meaning it has control privileges (read/write access to most operations) ￼. The handshake message includes scopes like operator.read, operator.write, etc., which the gateway uses to authorize actions ￼. By default, OpenClaw enables gateway authentication even on localhost, so the UI must supply the correct token or password or the connection will be rejected with an Unauthorized (1008) error ￼ ￼. The first time you open the dashboard, you typically include a ?token=<gateway token> in the URL (the CLI prints a tokenized link via openclaw dashboard) ￼ ￼. The UI will use that token in the connect call and, on success, store it in the browser’s localStorage for subsequent sessions ￼ ￼. This way, you don’t have to re-enter the token each time. The token is effectively an API key giving the Web UI client full control, so storing it locally (and using secure contexts) is important. The docs note that the Control UI should not be exposed publicly without protection, because it has admin powers (e.g. the ability to execute arbitrary commands via the AI) ￼. In practice, if you want to access the Web UI remotely, OpenClaw recommends using Tailscale VPN which provides a secure HTTPS tunnel and identity-based access, or using an SSH tunnel ￼ ￼. There is also a configuration to allow password auth or disable secure-device checks if you must run over plain HTTP on a LAN ￼ ￼ (not generally recommended). In a typical desktop setup (like a Tauri app embedding OpenClaw), you would run the gateway locally and point the Web UI to ws://127.0.0.1:18789 with the token, so everything stays on the user’s machine.

Local vs External Gateway: The Web UI can work with a local gateway (bundled) or connect to a remote gateway instance. In fact, the UI is flexible – since it’s just a web page that opens a WebSocket, you can configure it to target any reachable OpenClaw gateway. By default, when you run openclaw gateway on your machine, it also serves the UI at localhost:18789 and the WebSocket endpoint is ws://localhost:18789 (same host/port) ￼. In that scenario, the UI automatically connects to the same origin it’s served from. If you instead have an OpenClaw gateway running on another machine (say a cloud VPS), you can still use the Web UI by visiting the gateway’s URL (with appropriate networking set up). The UI will attempt the WebSocket connection back to that host. You’d need to provide the auth token (for example, via the URL param or by entering it in a prompt that the UI shows if unauthorized) ￼ ￼. The documentation provides guidance for remote access: using Tailscale’s Serve mode to expose the UI on a secure URL (so you can visit a HTTPS address and no token is needed if your Tailscale identity is allowed) ￼ ￼, or binding the gateway to a tailnet IP with a pre-shared token for manual access ￼. In any case, the communication architecture remains the same – one WebSocket to one gateway. The UI does not directly talk to multiple gateways at once (though you could open multiple browser tabs each to different gateway addresses if you ran separate instances). If running a gateway in “remote mode” (headless on a server), you might run the Web UI client in your browser locally and have it connect over the internet/VPN. The OpenClaw CLI can assist by printing an SSH tunnel command or opening the web link for you ￼ ￼. All of this means you can flexibly deploy OpenClaw in various topologies (local single-machine, client-server, etc.) and still use the same Web UI. In a desktop app context (like a Tauri app), one would typically embed the gateway and UI together: the app starts a local gateway process in the background and loads the Web UI front-end in a window (pointing to the local WebSocket). This is essentially what the OpenClaw macOS app does as well – it’s a SwiftUI wrapper around the same gateway WebSocket interface ￼ ￼.

State Management: Because the Web UI relies on the gateway for state, it queries for needed data and subscribes to events to reflect real-time changes. For instance, when you open the Sessions panel, the UI calls sessions.list and displays the result (session IDs, metadata) ￼. If you then use the chat to trigger a new session (/new), the gateway will create a new session entry and broadcast a session update event; the UI catches this (or simply re-queries sessions.list periodically or on certain actions) to update the session list UI immediately ￼. Similarly, for thinking/verbose mode toggles per session: OpenClaw allows you to enable verbose mode (where the AI’s chain-of-thought is visible) on a per-session basis. In the UI, there might be a toggle switch in the chat or sessions panel for “Show reasoning” which, when clicked, calls sessions.patch to set the verbose flag for that session ￼. Thereafter, the gateway will include the model’s reasoning messages in the events, and the UI will render them (perhaps in a special format). This illustrates how the UI both sends commands to modify state and listens for events to update the display, always keeping the browser view in sync with the underlying assistant state.

Data storage: All long-term data (conversations, skills, config, memory, etc.) is stored on disk by the gateway under the user’s home directory (e.g. ~/.openclaw/…). The Web UI itself stores only minimal info in the browser – mainly the auth token and some UI preferences in localStorage ￼. Even transcripts are not stored in the browser; if you refresh the page, it will fetch the conversation history anew from the gateway’s JSONL files via chat.history calls ￼. This means rebuilding the UI in another framework (like a Tauri app with React) can still leverage the same backend – you’d implement the same WebSocket protocol and draw the UI based on gateway responses, with no need to manage heavy local state. All critical data lives in the gateway, and the UI is essentially a real-time reflection of that data.

Support for Local Models (Llama.cpp Sidecar)

OpenClaw is model-agnostic and supports multiple AI model providers, ranging from OpenAI and Anthropic APIs to local runtime engines ￼. The default Web UI is designed to handle any model backend transparently – it will display the conversation and status regardless of which model is answering (Claude, GPT-4, a local LLaMA 2, etc.). If you want to use a local LLM (like llama.cpp) as the brain of your OpenClaw assistant, you can absolutely do so, and the Web UI will work with it without special modifications. Typically, you would run a local server that exposes an OpenAI-compatible REST API (for example, llama-cpp-python or Ollama), and then configure OpenClaw to use that as a custom model provider ￼. OpenClaw’s config allows adding a provider with an endpoint URL – once set (and perhaps an API key if needed), the gateway will route all model queries to your local llama.cpp server instead of the cloud. The UI will still show responses streaming, because the gateway streams data as it comes from the model. According to the FAQ, OpenClaw has direct support for Ollama (which simplifies running local LLaMA models), and in general “if your local server exposes an OpenAI-compatible API, you can point a custom provider at it” ￼. In practice, many users run a llama.cpp sidecar (for instance, via the llama-cpp REST server on port 8000) and configure OpenClaw’s model provider to use it – the assistant can then run fully offline. The Web UI will indicate the active model (often by name in the Overview or in the session info) and you might see slightly different performance (e.g. token streaming speed) but otherwise it behaves the same. The UI’s token counters will count tokens from your local model’s responses just as it would for OpenAI, since the gateway tracks those ￼.

One thing to note is that heavy use of a local model might cause slower response streaming or the bot “freezing” if the model is busy – the UI remains responsive in showing partial output, but if you find the browser UI lagging it’s usually due to the gateway/model saturating resources. The FAQ mentions that for very heavy workloads you can offload to another process or adjust settings ￼, but from a UI perspective no special action is needed: it will continue to receive chat events as the local model generates tokens.

In summary, all of OpenClaw’s Web UI features are model-backend-agnostic. Whether the assistant is powered by an OpenAI API or a llama.cpp instance on your machine, the Web UI’s design (sessions, chat, tools, etc.) remains the same. You just configure the gateway to use the desired model provider. The UI will show the model’s name and any provider-specific info in places like the health/debug panel, and you can switch models through the config panel (e.g. change default model and hit apply). If using local models, you might see the update in the UI that you’re now on “llama.cpp” or similar, confirming the sidecar is in effect. This pluggable architecture ensures you can rebuild the full feature set of OpenClaw’s UI in a desktop app and have it orchestrate both cloud models and local inference engines seamlessly.

Conclusion

The default OpenClaw Web UI is a full-fledged control center for your AI assistant, combining a chat interface with an entire suite of management tools. Its design centers on using the OpenClaw gateway as the brain and data store, with the UI as a real-time window and control panel. All features – from multi-session chat, to channel integrations, scheduled tasks, plugins, device management, configuration, and debugging – are accessible through a clean web interface communicating via WebSocket calls to the gateway ￼ ￼. In implementing a custom UI (e.g. in Tauri + React), one would replicate this behavior: maintain a WebSocket connection to the OpenClaw gateway, send JSON requests for actions, and handle incoming events to update the UI. The architecture is robust and extensible, allowing additional interfaces (mobile apps, terminal UI, etc.) to attach to the same gateway concurrently. By fully understanding how OpenClaw’s Web UI is structured – its features, functions, and data flow – you can confidently rebuild the experience in your own application, ensuring no loss of capability. OpenClaw’s “lobster way” is that everything can be controlled in one place; the Web UI exemplifies this by bringing together the assistant’s memory, skills, and operations into a single coherent dashboard for the user ￼ ￼.
