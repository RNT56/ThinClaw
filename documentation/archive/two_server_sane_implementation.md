# Scrappy: “Sane” Two‑Server Implementation (GPU Generation + CPU Embeddings)

This document is an implementation plan for the **sane** multi-server setup for Scrappy:

- **Server A (Generation / Chat)** runs on the **GPU** and is optimized for low-latency interactive chat.
- **Server B (Embeddings)** runs on the **CPU** and is optimized for RAG embeddings (document indexing + query embeddings) without competing for VRAM.

This is a **responsiveness + isolation** strategy, not a way to “double” a single GPU’s throughput.

---

## 1) Why two servers on one machine?

### Problem the design solves
RAG workflows create bursts of embedding work:
- indexing: thousands of chunk embeddings
- retrieval: query embedding per user message

If embeddings run on the GPU in the same runtime as generation, they can:
- reduce tokens/sec during chat
- increase queue wait time
- create a “stuttery” UX (chat feels sluggish during indexing)

### Sane solution
Move embeddings off the GPU entirely by running a **separate CPU-only embeddings server**, while keeping generation on GPU.

---

## 2) Non‑negotiable rule: Frontend never talks to llama-server

The frontend must **never** call `http://localhost:<port>` directly.

**Frontend → Tauri invoke() → Rust backend → server A/B**

Benefits:
- port and token never leak to JS
- no “port in use” errors
- unified request shaping and logging

---

## 3) Process model overview

```
[UI / WebView]
   invoke() commands
        |
[Rust backend (Tauri)]
  - SidecarManager (A + B)
  - Scheduler (interactive/background)
  - RAG pipeline (SQLite + FTS5 + sqlite-vec)
  - Reranker (ONNX, in-process)
        |
   -------------------------
   |                       |
[Server A: GPU chat]   [Server B: CPU embeddings]
  llama-server            llama-server
  /v1/chat/...            /v1/embeddings
```

## 4) Server responsibilities

### 4.1 Server A: Generation / Chat (GPU)
Used for:
- `/v1/chat/completions` streaming
- optional multimodal chat (Pixtral etc.)

Not used for:
- RAG embeddings
- bulk indexing

### 4.2 Server B: Embeddings (CPU)
Used for:
- `/v1/embeddings` for chunk embedding (indexing)
- `/v1/embeddings` for query embedding (retrieval)

Not used for:
- chat generation

---

## 5) Security and networking (authoritative)

For **both** servers:
- bind to `127.0.0.1` only
- choose a random free port at runtime
- require a random per-launch API key/token
- store port/token **in memory only** (never persist)

### 5.1 Pick a free port
Implementation pattern:
- bind `TcpListener` to `127.0.0.1:0`
- read the assigned port
- drop listener and spawn the server immediately (race window is tiny and acceptable for personal use)

### 5.2 Token
Generate a random 32+ byte token (base64 or hex). Inject it into every request as an auth header.

### 5.3 Backend proxy
Expose Tauri commands like:
- `chat_completion_stream(payload)`
- `embeddings(payload)`
- `index_document_job(payload)`
- `retrieve_context(payload)`

The Rust backend forwards to the correct server (A or B) using the in-memory port/token.

## 6) Model strategy for your daily models (Server A)

You plan to run these models daily:
- Pixtral (vision capable)
- Ministral
- GLM 4.7
- Qwen3 Coder (e.g., 30B GGUF)
- MiniMax-M2.1

### 6.1 Reality check for 16GB VRAM
- 8B–12B in Q4/Q5 is usually comfortable.
- 30B Q4 can be tight depending on context size and GPU offload settings.
- Multimodal (Pixtral + mmproj) adds overhead.

Scrappy should provide **Profiles** with safe defaults and auto-suggestions (already in your canon doc).

### 6.2 Recommended generation profiles (examples)
These are conceptual; the exact flags depend on the llama.cpp build and your GPU offload strategy.

- **Fast Chat (default)**
  - `np = 2`
  - moderate context (e.g., 4096–8192)
  - continuous batching enabled if it improves throughput under concurrency
  - reserve one slot for interactive lane

- **Max Responsiveness**
  - `np = 1`
  - smaller context
  - best for “I want the fastest typing experience”

- **Jobs Mode**
  - `np = 3–4` only if you know VRAM allows it
  - intended for batch tasks, not interactive

## 7) Embedding model strategy (Server B)

### 7.1 Why a dedicated embedding model
Using a chat model for embeddings is slower and less predictable. A small embedding model:
- is faster on CPU
- is “good enough” for retrieval
- reduces coupling between chat model changes and RAG index quality

### 7.2 Constraints
**Never compare embeddings generated by different embedding models.**
Therefore:
- each workspace must store `embedding_model_id` (name + version + file path hash)
- switching embedding model requires re-embedding/reindexing that workspace

### 7.3 Practical choices
Pick one embedding model you trust and keep it stable. (Scrappy can ship a “recommended embeddings model” entry in the Model Downloads page.)

If you want “text + code” RAG, choose an embedding model trained for both.

## 8) Launching the two servers

### 8.1 SidecarManager API
Implement a reusable sidecar manager:

- `spawn_generation_server(config) -> ServerHandle`
- `spawn_embedding_server(config) -> ServerHandle`
- `healthcheck(handle) -> Ready/NotReady`
- `restart(handle)` with backoff
- `shutdown(handle)`

`ServerHandle` stores:
- `child: std::process::Child`
- `base_url: http://127.0.0.1:<port>`
- `api_key`
- `started_at`
- `last_health_state`
- `stderr_tail` for diagnostics

### 8.2 Crash containment
If either server crashes:
- show a toast + diagnostics state
- restart automatically with exponential backoff
- enter “safe mode” if crash loop detected (for generation server)

---

## 9) Scheduler integration (two lanes)

Even though embeddings are on CPU, you still want prioritization.

### 9.1 Lanes
- **Interactive lane:** chat requests + query embeddings (needed for response)
- **Background lane:** ingestion embeddings (indexing), summarization jobs

### 9.2 Rules
- Query embeddings jump ahead of ingestion jobs.
- Ingestion yields when chat is active.
- Always reserve at least one generation slot for interactive lane (Server A).

---

## 10) RAG integration with two servers

### 10.1 Ingestion job (background)
1. Extract text (PDF/MD/TXT/etc.)
2. Clean headers/footers, normalize whitespace
3. Chunk (e.g., 300–800 tokens with overlap)
4. For each chunk:
   - call **Server B** `/v1/embeddings`
   - store chunk + embedding in SQLite
   - insert vector into `sqlite-vec` table (or store in embeddings table and rebuild vec index)
5. Mark document indexed; surface progress (“page 124/842”)

Performance tips:
- batch chunk embedding requests if your embedding endpoint supports it
- throttle concurrency (CPU can saturate quickly)
- allow partial availability: retrieval works once early chunks are indexed

### 10.2 Retrieval (chat-time)
1. User sends a message in a chat with RAG enabled
2. Rust backend computes query embedding via **Server B**
3. Run hybrid retrieval:
   - lexical: SQLite FTS5 BM25
   - vector: sqlite-vec KNN
4. Merge + dedupe; optionally boost items that appear in both
5. Rerank top candidates (ONNX cross-encoder in Rust) if profile enables it
6. Pack context under a strict token budget, with citations
7. Send augmented messages to **Server A** `/v1/chat/completions` with streaming

## 11) Multimodal (Pixtral) with Server A

If the selected model is vision-capable:
- Server A must be launched with the correct projector (`mmproj`) settings.
- Chat payload must include images in the format supported by llama-server.

### 11.1 Implementation contract
- Frontend:
  - accepts drag/drop or paste image
  - shows thumbnail preview
  - sends only file reference / attachment id to backend
- Rust backend:
  - stores image under app data
  - optionally downscales/compresses
  - base64 encodes and injects into the request payload
  - ensures the prompt references the image with the expected placeholder(s)

This keeps JS memory usage low and avoids shipping base64 across the JS boundary.

## 13) Failure modes and user experience

### 13.1 Embedding server down
- RAG features degrade gracefully:
  - show “Embeddings server restarting…”
  - allow chat without RAG
  - queue indexing jobs until server returns

### 13.2 Generation server down
- block sends
- show diagnostics and last stderr tail
- offer “Restart model” button
- safe mode fallback if crash loop

---

## 14) Performance expectations

What improves:
- chat remains responsive during indexing
- GPU throughput is not disrupted by embedding calls
- indexing can proceed in background without destabilizing generation

What does not magically improve:
- raw total throughput of generation (still one GPU)

This design primarily improves **responsiveness and predictability**.
