OpenClaw Agent Internal Execution Flow and Architecture

Message Lifecycle

When a user submits a message or task, it first enters the OpenClaw Gateway, a centralized WebSocket/HTTP control plane ￼ ￼. The Gateway routes the message to the appropriate agent session based on routing rules (e.g. which channel/account or peer it came from) and obtains a session key ￼. If an agent is already busy (a run active in that session), the new message is either queued or injected depending on the configured queue mode ￼. Otherwise, the Gateway creates a new agent run for that session. The message (along with any channel-provided context like recent chat history) is packaged into a standardized Body field for the agent, while the raw text is kept in a CommandBody for parsing directives ￼. The Gateway then issues an agent request into the agent runtime and immediately acknowledges the client with an accepted status and run ID ￼.

Once the agent run begins, the Gateway streams events back to the client in real-time. As the Large Language Model (LLM) generates the assistant’s reply, partial response tokens (deltas) are streamed as assistant events over the WebSocket ￼. These are buffered and chunked according to channel constraints (e.g. message length limits) before sending user-visible updates ￼ ￼. If the LLM’s reply consists of multiple “blocks” (e.g. paragraphs or code chunks), OpenClaw can optionally stream out each block as it completes (a feature called block streaming) instead of waiting for the full message ￼. Throughout the run, typing indicators and “presence” events may be emitted to let the user know the agent is working ￼. Once the agent has finished processing (either producing a final answer or encountering an error/abort), a final agent response event is sent with the completion status and a summary ￼. The Gateway then delivers the outbound reply to the original channel, handling any necessary chunking (splitting into multiple messages if too long) and adding prefixes or thread identifiers as configured ￼. Finally, if any messages were queued during the run, they will either trigger a new follow-up run or have been injected/handled per the queue policy.

Planning and Reasoning Flow

Prompt assembly is a critical first step once a message enters the agent runtime. OpenClaw constructs a comprehensive prompt that provides the LLM with all necessary instructions, context, and memory. This includes a system prompt defining the agent’s identity, rules, and capabilities (including descriptions of available tools) ￼. It also injects workspace “brain” files on the first turn of a session – for example, the agent’s SOUL.md persona and tone, IDENTITY.md (name/emojis), and any one-time BOOTSTRAP.md ritual – as well as persistent instructions or facts from AGENTS.md which acts as the agent’s long-term knowledge base or operating manual ￼. These files are trimmed if too large, or marked as missing if not present, to keep the prompt within size limits ￼. Next, the prompt assembler pulls in session context: the conversation transcript of recent user and assistant messages (pruned or summarized if needed) ￼, plus any relevant memory. OpenClaw distinguishes between the immediate user query and any historical context: recent messages from group chats may be prefixed with speaker labels, and channel-specific wrappers (like “[Chat messages since your last reply]”) encapsulate provided history ￼. The user’s new message is included last, after system and context, ensuring the LLM knows what to respond to.

With the full prompt ready, the agent runtime calls the configured LLM provider to generate a completion. OpenClaw supports multiple model backends (OpenAI, Anthropic, local models, etc.) via a provider abstraction ￼ ￼. The request is typically a streaming chat completion API call so that partial tokens can be handled. As tokens stream in, the runtime interprets the LLM’s output in an agentic format. Under the hood, OpenClaw uses a reasoning + acting prompt pattern (a form of chain-of-thought) so that the model can decide on tool use or provide answers. The system prompt encourages the model to “think” (possibly producing a hidden reasoning trace) and then either output an answer or a structured tool invocation. For example, the model might internally produce a reasoning like: Thinking… “I should search for X” then output an Action instruction for a tool. OpenClaw can optionally surface the model’s reasoning to the user (e.g. a “thinking” stream) or keep it hidden ￼. A special token like NO_REPLY can be emitted by the model when it intends to perform actions without immediately replying to the user (this token is filtered out and not shown) ￼.

Illustration of OpenClaw’s perception-action loop: a user prompt enters the agent’s planning phase, the LLM (agent core) decides on tools or answers, and tool results feed back into reasoning before a final response is produced. The agent’s internal policy and memory influence this loop. ￼ ￼

When the LLM indicates a tool invocation, the runtime pauses streaming and parses the tool call. OpenClaw defines a schema or format for tool instructions in the model’s output (for instance, a JSON block or a prefix like “Tool:” followed by tool name and parameters). The agent runtime recognizes this and extracts the tool name and arguments. It then looks up the tool in the available Skill registry. Tools can be built-in (file system read/write, shell commands, web browser automation, etc.) or user-installed “skills” (scripts in Python/Node, etc.), all exposed via a unified interface ￼. Before executing, the agent checks tool policy and session permissions. Core tools are always enabled by default, but certain dangerous actions might be gated (for example, apply_patch might be disabled unless explicitly allowed) ￼. Each tool runs in the context of the agent’s workspace directory (isolating file operations to that folder by default) and may be further sandboxed or have restricted access depending on config (read-only mode, no internet egress, etc., per security settings).

Tool Invocation Loop

OpenClaw’s agent operates in a cyclical planning-execution loop where the LLM’s plans (tool uses) are carried out and the results fed back for further reasoning. When the LLM outputs a tool instruction, the runtime invokes that tool and captures its output (stdout, return value, or any file produced). The execution is done asynchronously but serialized within a single turn – only one tool runs at a time per agent run, to maintain a clear thought -> action -> result sequence. As each tool starts and finishes, the Gateway emits tool events to clients with status updates ￼. For example, a tool_start event might indicate the agent is performing a web search, then a tool_end event carries the result. The agent runtime sanitizes tool outputs before using them: large outputs may be truncated or summarized, and binary data (images, etc.) might be replaced with references. This is both to conserve token space and avoid prompt injection from tool results ￼.

After a tool finishes, its result is injected into the LLM’s conversation context for the next round. Typically, OpenClaw will append a formatted observation like: “Tool X output:\n” back into the prompt, and then ask the LLM to continue. The LLM then consumes this and may decide to use another tool or finally answer the user. This loop of thought -> tool -> observation can repeat through multiple iterations until the LLM outputs a final answer with no further tool calls. Notably, the LLM can plan multiple actions in sequence: if the model’s response contains several tool calls one after another, the runtime will execute them one by one. After each tool call, the system checks if any new user message arrived (an interruption). If so and if the session is in an “interrupt/steer” mode, the agent will abort the remaining planned actions from the model’s output ￼. (Those remaining tools yield a “skipped due to queued message” result.) The new user query is then inserted, effectively diverting the agent’s course mid-thought. If no interruption, the loop continues.

Throughout the tool loop, OpenClaw ensures the user only sees appropriate output. Messaging tools (like a tool that sends a message or email) are tracked so that if they already produced a user-visible message, the agent doesn’t redundantly echo that content in its final answer ￼. The agent’s final response assembly will combine the assistant’s direct answer text with any important tool findings. For instance, if verbose mode is enabled, the agent may inline brief summaries of tool results (e.g. “(I searched the web and found XYZ)”) in the final answer ￼. Any internal reasoning content or the raw tool outputs that are not meant for the user are omitted or transformed. In the end, the agent produces a clean answer (which could be text, media, or a formatted result) that is sent to the user. The loop then ends for that turn, and the agent’s state (conversation log and any new memory) is updated.

Memory and Workspace Integration

Memory is a first-class concept in OpenClaw’s agent architecture, designed to give the agent long-term knowledge and context beyond the immediate chat history. Each agent has a dedicated workspace directory on disk which serves as its cognitive storage ￼. This workspace contains the aforementioned Markdown files (AGENTS.md, SOUL.md, etc.) that seed the agent’s initial context. More dynamically, the workspace has a memory/ folder where the agent accumulates daily notes. By default, OpenClaw uses two layers of memory files in the workspace: daily append-only logs (memory/YYYY-MM-DD.md for each day) and an optional permanent memory ledger (MEMORY.md) ￼. The daily files store a running journal of important events or facts in conversations, while MEMORY.md is meant for curated long-term facts (and is only loaded in private 1-on-1 sessions, not group chats, to avoid leaking personal info) ￼.

The agent is expected to write to memory as part of its reasoning. If the user says “remember X” or if the agent deduces something worth keeping, the correct behavior (guided by prompts) is to record it in a memory file rather than relying on the context window ￼. OpenClaw even automates this via a memory flush mechanism: when a conversation is getting too long and approaching the context limit, the system will trigger a special hidden turn (a pre-compaction step) prompting the model to save any important information to the memory files before older context is trimmed ￼. This prompt asks the model to output notes to the memory/YYYY-MM-DD.md file and respond with NO_REPLY if there’s nothing critical to save ￼. The agent runtime intercepts whatever the model “writes” in that turn and appends it to the memory file on disk (since it has file system write access via tools). This happens behind the scenes, and the user does not see the flush conversation (it’s a silent self-memory update) ￼. After compaction or between sessions, these memory files provide continuity – on a new session start, the agent will read yesterday’s and today’s memory log into context so it remembers recent discussions ￼.

OpenClaw also integrates vector-based memory search to help the agent recall facts even when not explicitly in the prompt. It maintains an embedding index of the content in memory files (and other opted-in documents) ￼. When the agent faces a complex query, it can invoke a memory search tool which performs an embedding similarity lookup against those notes. Relevant snippets can then be injected into the prompt (e.g. “Here are some notes: …”) for the LLM to use. By default, memory search is enabled and will use either a local embedding model or a cloud provider (OpenAI or Google Gemini) to generate embeddings for new memories ￼. This provides a form of long-term semantic memory. In addition to vector search, the agent can use classic keyword search on its transcripts or files (there are CLI commands and tools for searching past chats or files in the workspace). All these mechanisms – immediate session context, memory files, and vector recall – work together so the agent can incorporate past knowledge into current reasoning. The workspace, being the agent’s only allowed working directory, also means any files created or code written stay contained; for instance, if the agent uses a coding skill, it will create files under the workspace that persist for later use or inspection ￼. This design ensures persistency (the agent can refer to what it did previously) while maintaining a sandbox boundary around the agent’s activities.

Multi-Agent and Sub-Agent Flow

OpenClaw supports a multi-agent configuration where one Gateway can host multiple isolated agent personas. By default, it runs in single-agent mode (one agent ID, usually main), but you can define additional agents in the config ￼. Each agent has its own workspace, memory, and identity files, effectively creating separate “brains” under one roof ￼. In multi-agent setups, the Gateway uses binding rules to route inbound messages to the appropriate agent. These rules can match on the channel, account (e.g. which phone number or Slack workspace), sender, or group ID. For example, one might route personal WhatsApp messages to a “home” agent and work Slack messages to a “work” agent ￼ ￼. The routing is deterministic and uses the most specific binding that matches (peer ID first, then group/team, then account, etc., with a default agent as fallback) ￼. This allows multiple people to share one Gateway server, each with their own private AI assistant, or allows one user to segregate contexts (e.g. a coding agent vs. a personal assistant) ￼. The agents run in the same process but are logically isolated – they do not share sessions or memory unless explicitly allowed.

Notably, OpenClaw’s design doesn’t have a concept of an agent “spawning” sub-agents dynamically (like AutoGPT spawning new agents). All agents are predefined in the config. However, agents can delegate or communicate with each other in a limited way. There is an agent-to-agent messaging tool that, if enabled, allows one agent to send a message to another agent’s session via a tool call ￼. By default this is disabled (to prevent infinite loops or unintended back-and-forth), but an admin can allow certain agents to message each other by adding them to an allowlist ￼. When used, this essentially goes through the Gateway: one agent’s tool call is translated to an inbound message for the target agent, queued like any other user message. This could be used for delegation (one agent asks another for help on a task). Aside from that, multi-agent coordination is mainly achieved through the Gateway’s routing rather than spontaneous subagent creation. Each agent operates with its own concurrency control and context. For shared group chats, OpenClaw typically requires choosing one agent to handle the group (to avoid two agents responding at once), unless using a special broadcast mode where all agents see the message but only one actively responds ￼.

In summary, multi-agent in OpenClaw means multiple isolated agents under one Gateway, each with distinct personality and data, rather than a single agent splitting into sub-agents on the fly. Tasks can be delegated by passing messages between agents if configured. This isolation is a safeguard: one agent’s state or memory won’t bleed into another’s, and they even use separate API keys or model settings if specified (per-agent model config is supported). The Gateway ensures each inbound message is delivered to exactly one agent (the best-matched route) ￼, and each agent’s responses are sent out through the channel they came in on. This architecture, coupled with the workspace sandbox per agent, provides strong separation — akin to having multiple AI assistants living in the same house (Gateway) but with their own rooms and notebooks.

Execution Management and Safeguards

The OpenClaw agent runtime has numerous safeguards to ensure robust and safe execution. Concurrency is managed by design: each agent session processes only one message at a time, queueing any incoming messages that arrive mid-execution ￼ ￼. This command queue prevents race conditions in a session (no overlapping LLM calls for the same conversation). If multiple user messages come in rapid succession, a debounce setting can merge them or delay processing slightly, especially for things like typing bursts, to avoid the agent starting/stopping runs too frequently ￼. In multi-agent scenarios, different agents can run in parallel (since they are separate sessions/LLM calls), but they don’t share data except via explicit messages. The Gateway itself is single-instance and holds a lock on critical resources (like the WhatsApp web session, etc.) to prevent multiple gateway instances from clashing ￼ ￼.

OpenClaw implements timeouts and abort logic to handle stuck or slow operations. By default, an agent run request from a client will wait up to 30 seconds for a response (this is the client-side wait, configurable via agent.wait) ￼. Independently, the agent runtime enforces an absolute timeout (600 seconds by default, i.e. 10 minutes) for the entire turn’s processing ￼. If this runtime limit is hit, the agent run is aborted – the LLM is cut off and any running tool processes are terminated. Tools themselves may have their own timeouts; for example, web requests or shell commands are often executed via wrappers that can time out or get killed if they hang. There’s also a heartbeat/health monitor in the Gateway that can signal if the agent thread becomes unresponsive ￼ ￼. If a client disconnects or the WebSocket drops, the Gateway can cancel the corresponding agent run to free resources ￼.

Error handling and retry policy are built into the pipeline. If the LLM returns an error (e.g. API failure or content filter), the agent runtime will catch it and produce an appropriate assistant error message (unless the conversation context indicates it should suppress it) ￼. For tool errors, the agent doesn’t immediately fail the whole turn – the error is logged to the session and the model may get a chance to handle it. The final reply assembly will include a fallback message if a tool failed and the assistant didn’t already produce a user-facing message about it ￼. OpenClaw also has a retry-on-compaction mechanism: if a conversation has grown too long and the model’s answer seems to be getting cut off or not forthcoming, the system can trigger an automatic compaction (summary of older messages to free tokens) and then retry the last user query from scratch with the shortened context ￼. During such a retry, it clears any partial outputs or duplicated tool summaries to avoid double-sending content ￼. This improves reliability for lengthy sessions by giving the model a “second chance” with more context headroom.

Security safeguards are also in place at the execution level. Tools are executed in a constrained environment: by default, all file and shell commands operate in the agent’s workspace, not the entire filesystem, to limit scope ￼. Sandboxing settings (agents.defaults.sandbox) can further restrict tools – for instance running code in a docker or gVisor sandbox (in future or via extensions) ￼. OpenClaw’s config allows marking certain tools as “elevated” which require user approval each time (human-in-the-loop for risky actions) ￼, and untrusted content from the web can be sanitized before feeding to the LLM to mitigate prompt injection ￼. The agent’s credentials (API keys, etc.) are kept in an encrypted store or .env file and are not exposed to the LLM unless absolutely needed for a tool – preventing the model from simply printing secrets ￼. Furthermore, the Gateway can be secured behind an auth token and network restrictions (by default it binds to localhost) to prevent unauthorized external access ￼. These operational guardrails, combined with the controlled loop of thought and action, form a safety net around OpenClaw. In essence, the architecture ensures that the agent’s powerful abilities (reading emails, running code, controlling browser, etc.) are mediated through a careful funnel: all requests validated by the Gateway, all model decisions parsed by deterministic code, and all tool effects subject to timeouts, logging, and policy checks ￼ ￼. This way, the user gets a flexible and autonomous AI agent, but with transparent oversight and the possibility to intervene if something goes awry.
