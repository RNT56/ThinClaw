💬 Scrappy Chat Pipeline

Complete journey of a user message — from keypress to streamed response. Including text, multimodal, OCR/PDF, and all mode variations.

Main Chat Pipeline

The 10-stage path every message travels. Click any stage to expand details.

1User Sends MessageFrontend → IPC▼
Frontend invokes chat_stream Tauri command with a ChatPayload + IPC Channel<StreamChunk>.
  • ChatPayload: model, messages[], temperature, top_p, auto_mode, web_search_enabled, project_id, conversation_id
  • A generation mutex lock is acquired — only one generation at a time
  • The cancellation token is atomically reset to false
  • Messages are cloned for processing; user config is loaded from ConfigManager
2Provider Resolutionchat.rs → resolve_provider()▼
Reads selected_chat_provider from config and resolves into a ProviderConfig (kind, base_url, model, token, context_size).

Local MLX / vLLM

3Message Preprocessingchat.rs▼
Messages are cleaned and enriched. Multimodal images are base64-encoded, stale image history is stripped, and knowledge bits are gathered.

📝 Standard Text Message

  • Last message must be role "user"
  • Knowledge bits from UserConfig collected
  • Image history filtering: if not referencing images, strip prior image gen turns
  • has_context = true if any attached_docs or project_id present

🖼️ Multimodal (Images)

  • User messages with images[] → loaded via load_image_as_base64()
  • Content rewritten to JSON array: [{type:"text",...},{type:"image_url",...}]
  • Assistant image responses are never re-sent (context bloat protection)
  • If user references images, assistant gen text simplified to "[Assistant generated an image]"
4Effective Mode & Permissionschat.rs▼
Computes effective_auto_mode and ToolPermissions that control the entire downstream routing.
  • effective_auto_mode = auto_mode || web_search_enabled || has_context
  • enable_tools = effective_auto_mode — controls whether sandbox/tools are registered on the RigManager

Permissions When auto_mode=true

allow_web_search=✓ force_web_search=✗ allow_file_search=✓ allow_image_gen=✓

Permissions When web_search=true only

allow_web_search=✓ force_web_search=✓ allow_file_search=✗ allow_image_gen=✗

Permissions has_context only

allow_web_search=✗ force_web_search=✗ allow_file_search=✓ allow_image_gen=✗

Permissions All off (manual chat)

allow_web_search=✗ force_web_search=✗ allow_file_search=✗ allow_image_gen=✗ → No tools, Manual Mode

5RigManager + Orchestrator Initrig_cache.rs, orchestrator.rs▼
RigManager from cache (composite key hash). Orchestrator constructed with MCP config. Sandbox always built (MCP or local-only fallback).
  • Cache key = (kind, base_url, model, token, context_size, tools_enabled, knowledge_bits, model_family)
  • MCP sandbox if mcp_sandbox_enabled && mcp_base_url → full remote MCP tools
  • Fallback: build_sandbox_unconditional() → local host tools always available (web_search, rag_search, calculator, skills)
  • Persona instructions loaded from custom personas or built-in presets
6Orchestrator.run_turn() — Background Taskorchestrator.rs▼
Spawns Tokio task. Token counting (heuristic → precise fallback), auto-summarization at 60% context, document collection from history + current message.
  • Query extraction: Multimodal JSON array → extract text-only for RAG/tool queries
  • Token heuristic: ~4 chars/token; precise tokenizer only if within 80% of threshold
  • Auto-summarize trigger: token_count > context_window × 0.6 — summarize oldest 50% via LLM, prepend as system msg. Fallback: if tokenizer errors + history > 20 msgs.
  • Summarization: LLM call → condensed paragraph → ProviderEvent::ContextUpdate emitted → frontend updates history
  • Doc collection: All attached_doc IDs from history messages + current message gathered
⑦🔀 Decision: Manual vs Tool Mode▼
any_tools = allow_web_search || allow_file_search || allow_image_gen — this single boolean forks the entire execution.

Path A Manual Mode

  • No tools. Direct LLM call.
  • If docs/project attached → RAG retrieval inline (vector+FTS+rerank)
  • Up to 2 PDF visual previews injected for multimodal models
  • System prompt with persona + date + RAG context baked in
  • User query uses raw_content (preserves multimodal images)
  • Single stream_raw_completion() call → stream straight to frontend

Path B Tool / Auto Mode

  • Enters run_sandbox_loop()
  • System prompt includes tools description + Rhai usage instructions
  • ReAct loop: max 2 turns (force_search) or 5 turns (auto)
  • LLM can emit <rhai_code> blocks for tool execution
  • Sandbox has: web_search, rag_search, read_file, mcp_call, calculator, skills
  • Last turn: synthesis instruction injected, code tags stripped
8aManual Mode — RAG + Direct Streamorchestrator.rs :413–544▼
If docs or project attached: hybrid RAG retrieval (vector + FTS + RRF + reranker). Visual previews for PDFs. Then single LLM stream.
  • Emits <scrappy_status type="rag_search"> → frontend shows "Retrieving context…"
  • Text RAG: retrieve_context_internal() → vector search (scoped indices) + FTS → Reciprocal Rank Fusion → cross-encoder reranker → top 5 chunks
  • Visual previews: Up to 2 PDF first-page screenshots loaded as base64 images → injected as multimodal messages before user query
  • Context baked into system prompt: "[ATTACHED CONTEXT]:…"
  • Instruction: "Answer directly. Do NOT output <think> tags or simulate tool usage"
  • Single stream_raw_completion(conversation, None) → tokens flow to channel
8bTool Mode — ReAct Sandbox Looporchestrator.rs :580–990▼
Multi-turn reasoning. LLM can emit <rhai_code> blocks → executed in Rhai VM → results injected as <tool_result> → next turn.
  • System prompt: Persona + date + search rules (varies by force_search vs auto) + tools list + Rhai examples
  • Search rules: force_search = "ALWAYS search every query"; auto = "reply directly, only search for real-time data"
  • Available tools (conditionally registered): web_search(q), rag_search(q), read_file(path), generate_image(prompt), mcp_call(name, args), search_tools(q), calculator(expr), calculator_with_vars(expr, vars), run_skill(id, args), save_skill(id, script, desc)

🔄 Each Turn

Stream LLM → scan for <rhai_code> → if found: execute in sandbox → inject <tool_result> → loop. If no code: LLM answered directly → break.

🛑 Last Turn Guard

Synthesis instruction appended to last user msg. Any <rhai_code> tags are stripped. Content forwarded raw as natural language.

📊 Tool Output

Sandbox result → summarize_arbitrary_json() capped at 8000 chars → injected as <tool_result> user message

⚡ Status Events

OrchestratorStatusReporter → <scrappy_status> XML tags for tool activity, progress, thinking → streamed to frontend

9Stream Consumption & IPC Batchingchat.rs :555–703▼
mpsc::Receiver stream consumed. Tokens buffered (20 chars / 30ms) for IPC efficiency. Cancellation checked per chunk.

Content

Text tokens → batched → StreamChunk { content, done: false }

Usage

Token counts → StreamChunk { usage: Some(TokenUsage) }

ContextUpdate

Summarized history → StreamChunk { context_update } → frontend replaces stored msgs

Done / Cancel

{ done: true } or "[Stopped]" on cancellation. Buffer flushed before either.

✓Frontend Renders ResponseReact onmessage▼
Channel callback receives each StreamChunk, renders Markdown, parses <scrappy_status> tags, saves to SQLite on done.
  • <scrappy_status> tags → parsed into badges (searching, thinking, tool_call, progress)
  • First content token → transitions search status indicator to "done"
  • context_update → replaces conversation history in Zustand store
  • done: true → assistant message persisted to SQLite messages table

📄 Document Ingestion & OCR Pipeline

How documents get processed before they can be used as context in chat. This happens at upload time, not at query time.

1upload_document()rag.rs▼
File bytes arrive from frontend. Saved to $APP_DATA/documents/{id}_{filename}.
  • Random 8-char alphanumeric ID generated
  • Filename sanitized via Path::file_name()
  • Returns absolute path string for subsequent ingestion
2ingest_document()rag.rs▼
Reads file, SHA-256 hash for dedup. If hash exists and is "indexed" → reuse existing doc, update scope. Otherwise → full extraction.
  • Deduplication: same hash = same content → just update chat_id/project_id scope
  • New document: inserted into documents table with status "processing"
  • Calls extract_document_content() for actual content extraction
3🔀 Content Extraction Decisionrag.rs → extract_document_content()▼
File type detected. Text files read directly. PDFs use pdf_extract first, then garbage detection determines if Vision-OCR is needed.

📝 Text Files (.txt, .md, .rs, etc.)

  • String::from_utf8_lossy(buffer)
  • Null bytes filtered out
  • No OCR needed — direct to chunking

📑 PDF Files

  • Step 1: pdf_extract::extract_text() — native text layer
  • Step 2: Garbage detection — if <25% alphanumeric or >50KB but <1000 chars → scanned PDF
  • Step 3: If garbage/empty → Vision-OCR pipeline kicks in
  • Force OCR also triggered if pdf_extract fails entirely
4Vision-OCR Pipeline (PDFs only)rag.rs :109–223▼
Headless Chromium renders PDF pages → screenshots → local LLM Vision API transcribes each page. Up to 15 pages, 2-minute timeout.
  • Browser: chromiumoxide::Browser::launch() → headless Chromium, viewport 1280×1800px
  • Page rendering: Opens file://{path}, waits 1s for initial render
  • Per page: Screenshot (JPEG, 85% quality) → base64 → sent to local LLM via /v1/chat/completions
  • Prompt: "Transcribe all visible text. Maintain structure. Output ONLY text. If blank, output [empty]."
  • Scrolling: window.scrollBy(0, 1800) between pages, 800ms settle time
  • Early stop: If page returns [empty] after page 1 → stop scanning
  • Timeout: 120s overall → uses partial results if timed out
  • Preview: First page screenshot saved as $APP_DATA/previews/{hash}.jpg
5Content Assembly & Previewrag.rs :225–270▼
Final document content assembled. PDFs always get a visual preview generated (even without OCR). OCR text merged with native text.
  • If native text is very short (<100 chars) → OCR text replaces it entirely
  • If native text exists → OCR appended as "[OCR Supplemental Content]:…"
  • Preview generation: If preview doesn't exist and OCR wasn't used → separate headless Chrome session just for screenshot
6Chunking + Embedding + Indexingrag.rs :360–496▼
Content split into 1000-char chunks (100 overlap). Each chunk embedded via /v1/embeddings, stored in SQLite + HNSW vector index.
  • Chunking: Fixed 1000-char windows, 100-char overlap
  • Embedding: Local embedding sidecar (/v1/embeddings), concurrent 5-at-a-time via buffer_unordered(5)
  • Storage: chunks table (id, document_id, content, chunk_index, embedding blob)
  • Vector index: Scoped HNSW index via VectorStoreManager. Scope = Project(id), Chat(id), or Global
  • Rollback: If any chunk fails → delete document + all chunks. If index save fails → same.
  • Final: Document status → "indexed"
7RAG Retrieval at Query Timerag.rs → retrieve_context_internal()▼
When user sends a message with docs/project: hybrid search → RRF fusion → cross-encoder reranking → top 5 relevant chunks.
  • Vector search: Query embedded → search scoped HNSW index (project/chat + global), top-150
  • FTS search: SQLite FTS5 chunks_fts table, scoped to same project/chat, top-150
  • Reciprocal Rank Fusion (k=60): Merges vector + FTS scores
  • Cross-encoder reranker: Candle-based ONNX reranker rescores candidates, threshold -5.0 (or -10.0 for explicitly attached docs)
  • Special cases: File listing queries → return paths only. Filename mentions → return full doc content (up to 15k chars)
  • Vague queries ("summarize", "what is this"): First 3 chunks of each attached doc prepended as intro context

⚡ Auto Mode vs Manual Mode

Side-by-side comparison of what happens in each mode, and how flags combine.

?How the Mode is Determined▼
Three flags combine: auto_mode, web_search_enabled, and has_context. Any one of them being true activates tool mode.
  • auto_mode: User toggled the ⚡ auto button in the chat bar
  • web_search_enabled: User toggled the 🔍 search icon (legacy, maps to force search)
  • has_context: User attached documents or is in a project scope
  • Result: effective_auto_mode = auto_mode || web_search_enabled || has_context
  • If ALL three are false → Manual Mode (Path A)
  • If ANY is true → Tool Mode (Path B)
AManual Mode Path — No Tools▼
Simple, fast, direct LLM completion. Can still use RAG if docs are attached (though this technically also triggers tool mode via has_context).

📝 Plain Text Chat

System prompt (persona + date) → history → user message → single stream_raw_completion(). No tools, no sandbox.

🖼️ With Images

Same path, but user content is the multimodal JSON array. raw_content preserves the image_url entries for VLM servers.

  • System prompt explicitly says: "Do NOT output <think> tags or simulate tool usage"
  • Temperature: model default (not overridden, None)
  • This path is only truly reached if auto_mode OFF + no search + no docs/project
  • ⚠️ In practice, attaching a doc sets has_context=true which activates tool mode instead. So "Manual Mode with RAG inline" only fires for project context without auto_mode — a narrow edge case.
BTool / Auto Mode Path — Sandbox + ReAct▼
Full agentic pipeline. LLM can decide to call tools via Rhai code execution, or answer directly.

🔍 Force Web Search

web_search_enabled=true → System prompt: "ALWAYS SEARCH". User query wrapped: "SEARCH MODE ACTIVE: Call web_search ONCE…". Max 2 turns.

⚡ Auto Mode

auto_mode=true → All tools available (web, files, images). System prompt: "reply directly unless real-time data needed". Max 5 turns.

📄 Context Only

has_context=true → allow_file_search=true only. LLM can call rag_search() and read_file() but not web_search or image_gen.

🔌 With MCP Server

If mcp_base_url configured → search_tools() and mcp_call() also available. System prompt gets MCP-specific guidance.

  • Temperature: 0.1 during tool turns (more deterministic tool use)
  • The LLM can choose to NOT use tools and answer directly — the loop detects "no code" and breaks on turn 1
  • Tool results are JSON-summarized to max 8000 chars to prevent context overflow
  • calculator() and run_skill() are always available regardless of permissions
